## 1. 产品愿景

### 1.1 一句话定位

**让每一家餐厅都有一个不休息的 AI 合伙人。**

### 1.2 核心洞察

一家餐厅要活下来，老板需要同时扮演：厨师长、采购员、会计、营销经理、HR、客服、数据分析师。

传统 POS 只解决了"收银"这一个环节。各类 SaaS 工具分散在十几个系统里。老板每天工作 14 小时，大部分时间不是在做菜，而是在做运营——看报表、想促销、管会员、排班、对账、处理投诉。

**问题不是缺工具，而是缺人。**

一个老板无法同时做好所有运营决策，但也请不起运营总监、数据分析师、营销经理。

### 1.3 解决方案

FounderPOS 不是一个 POS 系统。它是一个 **AI 驱动的餐厅操作系统**。

```
传统 POS：   老板 → 用软件 → 管餐厅
FounderPOS： 老板 → 审批 AI 方案 → AI 管餐厅
```

如果每家餐厅有一个 AI，它能：

- 看懂所有经营数据
- 操作所有系统模块
- 对外代表餐厅接洽
- 但关键决策交给老板拍板

那么一个人就能管好一家餐厅，甚至十家。

### 1.4 四层架构

```
Layer 4 ─ Agent Identity（餐厅对外身份）
           每家餐厅 = 1 Agent + 1 Wallet
           对外开放：预定 / 包场 / 合作 / 询价
           Agent-to-Agent Protocol

Layer 3 ─ AI Operator（内部运营大脑）
           5 个顾问：菜单 / 营销 / 会员 / 经营 / 出品
           模式：AI 分析 → 生成方案 → 人类审批 → 执行

Layer 2 ─ MCP Tool Layer（工具执行层）
           覆盖所有业务域的标准化工具集
           统一 ActionContext（谁操作 / 为什么 / 谁批准）
           人类和 AI 走同一套工具，审计天然统一

Layer 1 ─ Transaction Foundation（交易底座）
           传统 POS 核心能力
           点餐 / 结账 / 会员 / 促销 / 报表 / 员工 / 班次
           稳定、可审计、独立可用
```

## 2. 市场背景

### 2.1 餐饮行业数字化现状

新加坡及东南亚餐饮行业正处于数字化转型加速阶段：

- 扫码点餐已成主流，顾客期望无需等服务员
- 外卖平台（Grab、Foodpanda）占营收比例持续提升，但抽成高达 25–35%
- 商场对商户 GTO 上报、CRM 对接的要求日益严格
- 用工成本上涨，人效压力迫使餐厅寻求自动化
- 消费者对会员权益、个性化推荐的期望越来越高

### 2.2 现有解决方案的局限

**传统 POS 系统**（如 EPOS、Square）：

- 只解决收银，不解决运营智能
- 无会员深度管理
- 数据在本地，无法支撑 AI 分析
- 与外部平台对接困难

**餐饮 SaaS 工具**（如美团收银、客如云）：

- 功能分散在多个系统
- 为中国市场设计，不适配新加坡监管和支付生态
- 无 AI 运营层，仍需人工操作每个模块
- 无 Agent 经济能力

**外卖平台自有系统**（Grab for Restaurant）：

- 只覆盖外卖场景，不覆盖堂食运营
- 数据归平台所有，商户无法获得完整经营洞察
- 无法做跨渠道统一会员

### 2.3 市场机会

新加坡约有 7,000+ 家餐饮门店，其中购物中心餐饮占比约 30%。这些门店特点：

- 租金高，对利润率敏感
- 商场要求多（GTO、会员、促销联动）
- 经营复杂（同时做堂食、自助、外卖）
- 人员流动大，依赖系统化运营

目前没有任何系统能一体化覆盖这类门店的全部需求，且没有 AI 运营层。**这是 FounderPOS 的核心机会窗口。**

## 3. 目标市场

### 3.1 主要目标市场

**新加坡购物中心餐饮商户**

典型代表：Aurora Jewel、Changi Airport 餐饮区、VivoCity 餐厅、ION Orchard 美食广场内的品牌餐厅。

特征：
- 坐落于购物中心，有商场 GTO 上报要求
- 同时经营多种业务形态
- 有会员运营需求
- 需要与外卖平台（Grab/Foodpanda）对接
- 对系统稳定性和审计能力要求高

### 3.2 支持的业务形态

| 业务形态 | 描述 | 关键需求 |
|---------|------|---------|
| **堂食单点（à la carte）** | 顾客入座 → 扫码/服务员点单 → 厨房出品 → 结账 | 桌台管理、KDS、分单支付 |
| **堂食自助餐（buffet）** | 顾客购买套餐包 → 自取或限量加点 → 计时结束 | 套餐层级、计时器、加点规则、超时费 |
| **平台外卖** | Grab/Foodpanda 接单 → 厨房出品 → 骑手取餐 | 平台对接、订单同步、KDS 路由 |
| **自有外卖渠道** | 顾客通过自有小程序/链接下单 → 配送 | 独立订单入口、LalaMove 对接 |
| **外带（takeaway）** | 顾客到店下单打包 | 快速收银、无桌台流程 |

同一家门店可以同时经营以上全部业务形态。

### 3.3 次要目标市场（扩展阶段）

- 茶饮连锁（如 Koi、LiHo）
- 咖啡独立门店
- 快餐和轻食（需要快速收银）
- 烘焙和零售复合型门店

## 4. 核心价值主张

### 4.1 三层价值栈

```
┌─────────────────────────────────────────────────────────┐
│  Layer 3: Restaurant Agent + Wallet                      │
│  餐厅有对外身份 → 接受预定、包场询价、供应商报价         │
│  Agent 经济：餐厅成为可交互的 Agent 节点                 │
│                                                          │
│  Layer 2: AI Operator（5 顾问）                          │
│  菜单顾问 / 营销顾问 / 会员顾问 / 经营顾问 / 出品顾问    │
│  AI 分析 → 生成方案 → 人类审批 → 自动执行               │
│                                                          │
│  Layer 1: 交易底座（传统 POS 核心）                      │
│  点餐 / 结账 / 会员 / 促销 / 报表 / 库存 / 员工          │
│  稳定、可审计、独立可用                                  │
└─────────────────────────────────────────────────────────┘
```

### 4.2 AI 五顾问体系

| 顾问角色 | 感知范围 | 典型建议示例 |
|---------|---------|------------|
| **菜单顾问** | SKU 销量、毛利率、退菜率、出菜时间 | "芝士焗饭连续 3 周毛利为负，建议下架" |
| **营销顾问** | 客流趋势、时段分布、促销效果、竞对活动 | "周三下午客流低 40%，建议推满 $50 减 $8" |
| **会员顾问** | 消费频次、储值余额、流失风险、积分到期 | "287 个会员 30 天未消费，推送召回充值活动" |
| **经营顾问** | 日报、周报、毛利分析、人效、异常预警 | "本周毛利下降 5%，主因牛肉成本涨 12%" |
| **出品顾问** | 平均出菜时间、退菜原因、高峰瓶颈 | "午高峰出菜超 15 分钟的菜品 3 道，建议简化制作流程" |

**AI 运作五步循环：**

```
Sense  → 通过 MCP Tools 持续感知经营数据
Think  → LLM 分析数据，识别问题和机会
Propose → 生成具体建议方案（含草案内容）
Approve → 推送给老板审批（低风险可自动执行）
Act    → 通过 MCP Tools 执行，记录 ActionContext
```

### 4.3 Restaurant Agent + Wallet

每家餐厅拥有对外身份和钱包，成为 Agent 经济的参与节点：

```
Restaurant Agent "Aurora"
├── Identity：名称 / 地址 / 菜系 / 风格 / 营业时间 / 认证信息
├── Wallet：收款 / 分账 / 结算 / 预算控制 / 智能合约分润
├── Capabilities：可接预定 / 可包场 / 可外卖 / 可团购 / 可联合采购
└── Protocol：对外暴露的交互接口和响应规则
```

## 5. 用户角色与权限

### 5.1 角色体系

| 角色 | 中文 | 核心权限范围 |
|------|------|------------|
| **Customer** | 顾客 | 扫码点单、查看订单、支付、积分查询、会员中心 |
| **Cashier** | 收银员 | 前台点单、收银结账、订单修改（权限内）、找零打印 |
| **Waiter** | 服务员 | 开台、叫单、催菜、桌台状态更新、换台、拼台 |
| **Kitchen** | 厨师/厨房员工 | KDS 查看出品队列、确认出品、标记延误、叫取餐 |
| **Store Manager** | 店长 | 全门店数据访问、员工管理、班次安排、促销审批、退款审批 |
| **Area Manager** | 区域经理 | 跨门店数据查看、巡店记录、绩效对比、政策下发 |
| **Finance** | 财务 | 结算报表、对账、发票、成本数据、税务数据 |
| **Marketing** | 营销 | 会员管理、促销配置、券码发放、活动分析 |
| **Owner** | 门店老板 | 全数据权限、AI 审批、战略决策、商业配置 |
| **Shareholder** | 股东 | 只读经营数据、RWA 数据、分红记录 |
| **Platform Admin** | 平台管理员 | 商户管理、系统配置、平台报表、技术支持 |

### 5.2 权限设计原则

- 基于 RBAC（角色访问控制）+ 资源+动作二维矩阵
- 敏感操作（退款、价格修改、折扣超限）需上级角色审批
- AI 执行的操作在 ActionContext 中记录审批链
- 所有写操作留审计日志，不可删除

## 6. SKU 三层模型

餐饮的一切业务动作都围绕 SKU 进行。SKU 必须同时建模三个层面：

```
┌──────────────────────────────────────────────────────────┐
│                         SKU                               │
│                                                           │
│  Layer 1: 顾客侧（展示 + 销售）                            │
│  ├── 名称（多语言）、图片、描述、过敏原标注                  │
│  ├── 售价（基础价 + 多场景覆盖）                            │
│  ├── 修饰符（辣度、汤底、加料、规格选项）                   │
│  ├── 推荐标签（主推、热销、新品、特惠）                     │
│  ├── 可售渠道（堂食 / 外卖 / 自助餐 / 外带）               │
│  └── 可售时段（早茶 / 午市 / 下午茶 / 晚市 / 夜宵）        │
│                                                           │
│  Layer 2: 厨房侧（制作 + 出品路由）                        │
│  ├── 归属工作站（主厨房 / 凉菜站 / 饮品站 / 点心站）        │
│  ├── 打印路由（打印机 IP / KDS 编号）                      │
│  ├── 出品备注模板（标准出品说明）                           │
│  ├── 预估制作时间（用于 ETA 计算）                         │
│  └── 优先级标记（VIP 桌优先 / 快出菜标记）                 │
│                                                           │
│  Layer 3: 库存侧（成本 + 消耗扣减）                        │
│  ├── 成本价（用于毛利计算）                                 │
│  ├── SOP 配方（消耗哪些原料、多少量）                      │
│  ├── 单位换算规则                                          │
│  └── 是否需要库存实时扣减                                  │
└──────────────────────────────────────────────────────────┘
```

### 6.1 多渠道定价矩阵

同一 SKU 在不同场景下可以有不同售价：

| 维度 | 示例 |
|------|------|
| **按门店** | 主店价 $28，机场店价 $35 |
| **按渠道** | 堂食价 $28，外卖价 $32（含包装费） |
| **按会员等级** | 普通价 $28，Gold 会员价 $25 |
| **按时段** | 午市价 $28，晚市价 $32 |
| **按套餐** | 单点价 $28，自助餐套餐内加购价 $20 |

定价优先级（从高到低）：时段价 > 会员等级价 > 渠道价 > 门店基础价

## 7. P0 核心需求（上线必须）

### 7.1 桌台与扫码点餐

**桌台管理：**

- 门店地图：可配置桌台布局（区域 + 桌号 + 桌台容量）
- 桌台状态：空闲 / 就坐 / 点单中 / 出品中 / 待清台
- 开台：扫描桌台 QR 码或前台手动开台，支持人数录入
- 换台：桌台数据（订单 + 成员）整体迁移到新桌
- 拼台：两张桌合并为一个订单
- 加位：已开台桌台修改就坐人数

**顾客扫码点餐：**

- 每张桌台生成唯一 QR 码
- 顾客扫码进入点单界面（微信小程序 / H5）
- 支持多人共同点单（同一桌不同顾客分别扫码，追加到同一订单）
- 扫码点单和前台点单统一归到同一桌台订单
- 顾客可在手机查看订单状态（已下单 / 制作中 / 已出品）
- 不强制要求登录，但扫码点单时可引导绑定会员

### 7.2 菜单系统

**三种菜单模式：**

#### 7.2.1 堂食单点模式（à la carte）

- 菜单按分类展示：前菜 / 主菜 / 汤 / 甜品 / 饮品
- 支持加料选项（修饰符）
- 支持备注输入（不加香菜 / 少辣）
- 支持加单（已出部分菜后继续点菜）
- 支持菜品下架、售罄标记（实时生效）

#### 7.2.2 堂食自助餐模式（buffet）

**套餐层级（Package Tiers）：**

- 支持多个套餐选项（如：成人套餐 $38、儿童套餐 $18、高级套餐 $58）
- 每个套餐配置：价格、包含内容说明、是否支持加点
- 自助餐专属菜单：与单点菜单隔离，自助餐顾客只看自助餐菜单
- 支持自助餐顾客额外单点套餐外菜品（按单点价计费）

**计时器（Timer）：**

- 开台时启动计时
- 可配置用餐时长限制（如：90 分钟）
- 到时前 15 分钟系统提醒服务员
- 超时自动追加超时费或提示结账

**超时费用（Surcharges）：**

- 可配置超时收费规则（如：每 30 分钟加收 $5）
- 自动计入账单

**加点规则：**

- 可配置每桌每人可加点的轮次限制
- 可配置某些菜品每人只能点一份
- 自助餐内加点菜品追加到厨房 KDS 队列

#### 7.2.3 外卖模式（delivery）

- 独立外卖菜单（可与堂食菜单部分共用，但独立配置可售性和价格）
- 支持最低起送金额配置
- 支持配送费配置
- 支持外卖专属套餐和组合
- 平台外卖（Grab/Foodpanda）接单后自动注入系统，进入 KDS 队列
- 自有外卖渠道支持顾客在线下单

### 7.3 厨房显示系统（KDS）与出品路由

**KDS 工作站：**

- 支持多个物理 KDS 屏幕，每个屏幕对应一个工作站
- 工作站类型：主厨房、凉菜站、饮品站、点心站、前台备餐台
- 每道菜在 SKU 层配置归属工作站，下单后自动路由到对应 KDS

**KDS 展示内容：**

- 订单号 + 桌台号（或外卖单号）
- 菜品名称 + 修饰符选项 + 备注
- 下单时间 + 等待时长（超时变红高亮）
- 出品顺序优先级

**出品流程：**

- 厨师点击确认出品 → 状态更新 → 通知服务员取餐
- 支持部分出品（一桌的菜分批出）
- 支持催菜：服务员标记催菜，KDS 高亮显示
- 支持退菜：从 KDS 撤除，记录退菜原因

**打印路由：**

- 下单后自动打印厨房小票（按工作站分别打印）
- 支持同时打印多张（不同工作站各自接收对应菜品）
- 支持备份打印（KDS 故障时自动切换到打印机）

### 7.4 会员与 CRM

#### 7.4.1 会员注册与识别

- 注册方式：扫码点餐绑定 / 前台开单绑定 / 主动注册
- 识别方式：手机号 / 会员码 / NFC 卡 / 扫码（未来）
- 会员资料：姓名、手机号、生日、邮件、偏好标签

#### 7.4.2 积分体系

- 积分规则：消费金额按比例获得积分（可按渠道/时段/活动差异化配置）
- 积分用途：兑换现金抵扣、换礼品、换升级
- **批量过期（Batch Expiry）：** 积分按自然年或自然季度统一过期，非按每笔消费计时
- **FIFO 扣减（先进先出）：** 消费时优先扣减最早获得的积分
- 积分到期前 30 天推送提醒
- 积分记录完整可查（获取原因 + 扣减原因 + 过期原因）

#### 7.4.3 储值账户（Stored Value）

- 顾客充值：支持现金、卡、QR 充值
- 充值赠送：配置充值满 X 送 Y（可按会员等级差异化）
- 余额消费：结账时可用余额抵扣
- 余额查询：顾客自助查询
- 退款规则：储值款项不予退款（或按门店政策配置）

#### 7.4.4 优惠券（Coupons）

- 类型：满减券、折扣券、免费菜品券、赠品券
- 发放方式：手动发放 / 系统自动触发 / 营销活动批量发放
- 使用规则：有效期、适用渠道、适用菜品范围、单次上限、叠加规则
- 核销记录：每张券的核销历史可追溯

#### 7.4.5 转介绍（Referrals）

- 每个会员生成专属推荐码
- 被推荐人通过推荐码注册且首次消费后，推荐人获得奖励（积分/券/储值）
- 被推荐人同步获得欢迎奖励
- 推荐关系链可查询

#### 7.4.6 会员等级（Tiers）

- 支持自定义等级（如：Silver / Gold / Platinum）
- 升级规则：按累计消费金额 / 消费次数触发
- 等级权益：专属折扣、积分倍率加成、优先接待、专属菜品
- 降级规则：可配置（如：一年内消费不达标降级）

### 7.5 结算系统

**支持支付方式：**

| 支付方式 | 说明 |
|---------|------|
| **现金** | 收款 + 找零，记录实收金额 |
| **银行卡（DCS）** | 对接 DCS Payment SDK，支持刷卡/插卡/NFC |
| **QR（VibeCash）** | 对接 VibeCash 扫码支付 |
| **积分抵扣** | 消费积分按规则折算为抵扣金额（FIFO） |
| **储值余额** | 使用账户余额支付 |
| **混合支付** | 以上多种方式组合（如：储值 $50 + 现金 $20） |

**结账流程：**

- 打印预账单 → 顾客确认 → 选择支付方式 → 完成支付 → 打印收据
- 支持分单结账（一桌多人 AA 分摊）
- 支持整桌结账
- 退款：支持原路退款，需记录退款原因，需上级授权

**发票/收据：**

- 电子收据发送至顾客手机/邮件
- 商场 GTO 上报格式支持

### 7.6 库存管理

#### 7.6.1 基础数据

- **食材档案：** 原料名称、单位、供应商、最低库存预警量
- **SOP 配方：** 每个 SKU 消耗哪些食材、多少量（支持嵌套半成品）
- **单位换算：** 如：1 千克 = 1000 克，1 箱 = 24 瓶

#### 7.6.2 采购管理

- 供应商管理：供应商档案、联系人、常购品目、付款条款
- 采购订单：手动创建 / 系统根据库存预警自动建议
- 收货确认：收货数量录入，自动入库
- 采购记录：按供应商 / 时间 / 品类可查

#### 7.6.3 库存扣减

- **实时扣减：** 每笔订单出品后，按 SOP 配方自动扣减对应食材库存
- **批量扣减：** 支持按日/班次统一处理
- **手工调整：** 支持盘点后的库存修正，记录原因

#### 7.6.4 预警与盘点

- 低库存预警：食材库存低于阈值时推送通知给店长/采购
- **盘点（Stocktake）：** 周期性盘点流程，录入实际库存，系统自动计算差异
- 差异报告：盘点差异自动标记（高差异项需要解释原因）

#### 7.6.5 损耗管理

- 记录食材损耗原因（过期 / 操作失误 / 自然损耗）
- 损耗统计报表：按时段、品类、员工维度统计
- 损耗成本自动计入经营成本

### 7.7 报表系统

| 报表类型 | 核心指标 | 频率 |
|---------|---------|------|
| **日报** | 营收、客单价、订单数、毛利、支付构成、渠道占比 | 每日自动生成 |
| **销售报表** | SKU 销量排行、SKU 毛利率、退菜率、时段分布 | 按需 / 周 |
| **会员报表** | 新增会员、活跃会员、储值总额、积分流通量、流失预警 | 按需 / 周 |
| **时段报表** | 按小时/时段的客流、营收、客单价对比 | 按需 |
| **渠道报表** | 堂食 vs 外卖 vs 外带，各渠道收入、成本、利润 | 按需 / 周 |
| **库存报表** | 食材消耗量、损耗量、采购金额、库存周转率 | 按需 / 周 |
| **员工报表** | 人效（营收/人）、班次覆盖率、overtime | 按需 / 月 |

AI 经营顾问基于以上数据自动生成日报摘要，推送给老板。

### 7.8 外部集成

#### 7.8.1 商场系统

- **商场 GTO（Gross Turnover）：** 每日/每月自动上报营业额，符合商场要求格式
- **商场 CRM 对接：** 支持商场会员点数互联（顾客在餐厅消费可累积商场积分）
- 支持商场统一促销活动参与（如商场购物节联动活动）

#### 7.8.2 外卖平台

- **Grab for Restaurant：** 菜单同步（FounderPOS 菜单变更自动同步到 Grab）、订单接入（Grab 新订单自动进入 KDS 队列）、状态回传（出品确认后通知骑手）
- **Foodpanda：** 同上
- 订单来源标记：所有外卖平台订单在报表中区分渠道
- 佣金记录：平台抽成自动计入各渠道成本

#### 7.8.3 预订系统

- **Google Reservation：** 通过 Google 商家页面接收预约，自动同步到门店桌台管理
- 预约确认：自动回复确认邮件/短信

#### 7.8.4 评价系统

- **Google Review：** 结账后自动触发 Google Review 邀请（短信/邮件）
- 评价监控：聚合 Google Review 数据，异常差评推送通知

### 7.9 推荐与智能展示

#### 7.9.1 菜单推荐

- **主推菜（Featured Items）：** 店长手动标记 + AI 建议（基于毛利率和销量）
- **加购推荐（Upsell）：** 点单时根据已选菜品推荐搭配（如：点了牛排推荐红酒）
- **热销榜（Trending）：** 实时基于当日/本周销量动态更新
- **会员偏好推荐：** 已知会员历史点单数据，优先展示其偏好菜品

#### 7.9.2 展示逻辑

- 推荐优先级：会员个性化 > 时段特推 > 热销榜 > 主推
- 每次菜单加载时，AI 根据时段、渠道、会员信息动态排序

### 7.10 时段菜单与多渠道定价

#### 7.10.1 时段菜单（Time-slot Menus）

- 菜单按时段激活：如早茶菜单（08:00–11:00）、午市菜单（11:00–14:30）、晚市菜单（17:00–22:00）
- 不同时段可展示不同菜品组合和价格
- 时段菜单自动切换，无需人工干预
- 支持特殊日期配置（节假日菜单覆盖）

#### 7.10.2 多渠道定价（Multi-channel Pricing）

同一 SKU 在以下维度可以配置独立价格：

```
价格决策矩阵：
  ├── 门店维度：主店 / 分店 / 机场店
  ├── 渠道维度：堂食 / Grab / Foodpanda / 自有外卖 / 外带
  ├── 会员等级维度：Guest / Silver / Gold / Platinum
  └── 时段维度：午市 / 晚市 / 节假日
```

优先级规则：时段价 > 会员等级价 > 渠道价 > 门店基础价

#### 7.10.3 渠道归因与佣金分润（Channel Attribution）

- 每笔订单记录来源渠道（堂食 / Grab / Foodpanda / 自有外卖 / 外带）
- 外卖平台佣金自动按比例从渠道收入中扣除
- 报表中展示各渠道净利润（扣除佣金后）
- 支持自定义渠道合作分润规则（如：与商场会员系统的营收分成）

## 8. P1 增强需求（上线后第一轮）

### 8.1 排队/等位系统（Queue / Waitlist）

- 顾客通过二维码或前台加入等位队列
- 系统展示当前等位人数和预计等待时间
- 轮到时自动发送短信/微信通知
- 超时未到自动跳过（可配置保留时长）

### 8.2 自有外卖 + LalaMove 配送

- 顾客通过自有小程序下单外卖
- 订单进入 KDS 后，厨房确认出品时系统自动呼叫 LalaMove 骑手
- 骑手接单状态实时同步到前台
- 支持多家骑手平台比价（LalaMove / Lalamove / 其他）
- 配送状态推送给顾客

### 8.3 营销自动化（WhatsApp / SMS / Email）

- 触发条件配置：生日 / 积分到期 / 消费 N 天未回头 / 首次消费后 / 充值成功后
- 消息模板：支持变量填充（姓名、积分余额、优惠码）
- 渠道优先级：WhatsApp > SMS > Email
- 发送记录与打开率统计
- A/B 测试支持（不同话术测试点击率）

### 8.4 评价与反馈系统

- 结账后自动触发满意度调查（1–5 星 + 文字）
- 差评（1–2 星）触发内部预警，自动通知店长
- 好评（4–5 星）引导到 Google Review
- 反馈汇总报表：按菜品、服务、环境维度统计

### 8.5 智能排班

- 基于历史客流数据和预订数据，AI 建议每天所需人手配置
- 店长可在系统中排班、调班、批准假期申请
- 员工通过 App 查看排班、提交换班请求
- 自动计算加班费、计算工时成本

### 8.6 CCTV AI 监控

- 对接门店 CCTV 系统，AI 分析视频流
- 自助餐台检测：餐台空盘 / 食物不足 / 垃圾堆积 → 自动推送通知
- 异常行为检测：可疑行为标记（需人工确认）
- 顾客动线分析：热力图，优化排队和取餐区域布局

## 10. 商业模式

### 10.1 收费分层

| 层级 | 产品形态 | 定价策略 | 目标 |
|------|---------|---------|------|
| **基础 POS** | 交易底座 + 基础 KDS + 基础报表 | 低价/免费 | 快速获客，建立数据基础 |
| **AI 功能订阅** | 五顾问 + 营销自动化 + 智能推荐 | 按月订阅 $X/月 | 主要收入来源 |
| **Agent 经济** | Restaurant Agent 对外交互 + Wallet 结算 | 按交易笔数抽成 | 规模化后的增量收入 |
| **RWA** | 经营数据上链 + 发债服务 | 融资金额的 X% 服务费 | 长期高价值收入 |

### 10.2 收费模式细节

**基础 POS（低价策略）：**

- 月费低于市场主流 POS（如 EPOS 约 $200/月），甚至提供免费基础版
- 降低切换成本，让商户愿意迁移
- 硬件（KDS 屏幕、打印机）可租赁或按硬件成本出售

**AI 订阅（核心收入）：**

- AI 五顾问完整功能：约 $150–300/月/门店
- 营销自动化附加包：约 $50–100/月
- 高峰期消费，订阅价值可覆盖多倍月费

**Agent 经济（规模收入）：**

- 包场交易：收取 1–2% 服务费
- 联合采购：收取采购金额 0.5% 平台费
- 导流成交：按成交金额收取 3–5% 佣金

**RWA（战略收入）：**

- 帮助餐厅基于经营数据发行收益权凭证
- 收取融资金额 2–3% 的服务费
- 为平台积累最高质量的商户（有融资能力证明）

### 10.3 护城河分析

```
飞轮效应：
  更多餐厅加入
       ↓
  更多匿名经营数据
       ↓
  AI 顾问建议更准确
       ↓
  商户经营效果更好
       ↓
  商户愿意续费 + 推荐
       ↓
  更多餐厅加入（循环）
```

数据网络效应：1000 家餐厅的数据让每家店的 AI 更聪明，竞争对手无法复制。

## 11. 竞争差异化分析

### 11.1 竞争格局

| 维度 | 传统 POS（EPOS/Square） | 餐饮 SaaS（客如云/美团） | 外卖平台系统（Grab） | **FounderPOS** |
|------|----------------------|----------------------|-------------------|---------------|
| **核心定位** | 收银工具 | 管理工具集合 | 外卖运营工具 | AI 运营操作系统 |
| **覆盖场景** | 堂食收银 | 堂食 + 简单外卖 | 仅外卖 | 全场景（堂食/自助/外卖/外带） |
| **AI 能力** | 无 | 弱（仅数据分析） | 无 | 核心差异（5 顾问 + 执行） |
| **用户操作量** | 高（所有操作人工） | 中（部分自动化） | 高 | 低（只做关键审批） |
| **数据归属** | 商户本地 | 平台内闭环 | 归平台 | 商户完全拥有 |
| **跨店能力** | 弱 | 中 | 弱 | 强（Agent 网络） |
| **Agent 经济** | 无 | 无 | 无 | 核心路线图 |
| **RWA 能力** | 无 | 无 | 无 | 长期差异化 |
| **本地化适配** | 有新加坡版本 | 中国设计，难适配 | 新加坡原生 | 新加坡原生 |
| **商场集成** | 有 GTO 支持 | 不支持 | 不支持 | 全面支持 GTO + CRM |

### 11.2 核心差异化主张

**对比传统 POS（EPOS/Square）：**

> FounderPOS 不只是一个收银工具，而是让 AI 替你操作系统。传统 POS 你是用户，FounderPOS 你是老板，AI 是员工。

**对比餐饮 SaaS：**

> SaaS 给你工具，你还是要自己用。FounderPOS 的 AI 主动分析、主动建议、主动执行，你只需要在关键决策上点「批准」。

**对比外卖平台系统（Grab for Restaurant）：**

> Grab 只帮你管外卖，但你 70% 的营收来自堂食。FounderPOS 覆盖你的全部经营，而且数据是你的，不是平台的。

### 11.3 进入壁垒

对于竞争对手来说，难以复制的核心要素：

1. **数据飞轮：** 需要大量餐厅数据训练 AI 模型，后来者无法快速复制
2. **MCP 工具体系：** 标准化的 AI 可操作工具层，需要深度领域知识积累
3. **Agent 经济协议：** 先行建立 Restaurant Agent 标准，形成协议护城河
4. **商户信任：** POS 数据涉及核心财务，商户迁移成本高，先进入者优势明显

## 12. 成功指标

### 12.1 获客指标

| 指标 | P0 目标（上线 3 个月） | P1 目标（上线 12 个月） |
|------|---------------------|---------------------|
| 激活门店数 | 5 家（Aurora 等早期客户） | 50 家 |

### 12.2 产品使用指标

| 指标 | 目标 |
|------|------|
| AI 建议采纳率 | ≥ 60%（老板批准 AI 方案的比例） |
| 日活跃 KDS 使用率 | ≥ 90%（有出品的门店当日使用 KDS） |
| 会员渗透率 | ≥ 30%（消费者中绑定会员的比例） |
| 外卖平台订单自动处理率 | ≥ 95%（Grab/Foodpanda 订单无需人工干预） |

### 12.3 商业指标

| 指标 | 目标 |
|------|------|
| 月度经常性收入（MRR）增长率 | ≥ 15%/月（前 12 个月） |
| AI 订阅渗透率 | ≥ 70%（基础 POS 用户中升级 AI 订阅比例） |

- 毛利率提升 ≥ 3%（通过 SKU 优化和成本控制）

- 老板每周在运营决策上节省 ≥ 5 小时
