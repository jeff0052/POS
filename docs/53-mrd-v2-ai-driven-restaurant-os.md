# Market Requirements Document V2: AI-Driven Restaurant Operating System

Date: 2026-03-26

---

## 1. Document Purpose

本 MRD 定义 FounderPOS 从"餐饮 POS 工具"升级为"AI 驱动的餐饮操作系统"的市场需求、目标客户、核心价值主张和商业模式。

本文档是 `docs/02-market-requirements-document.md`（传统 POS MRD）的升级版，涵盖 AI Operator、Restaurant Agent、Wallet 和 Agent Network 等新概念。

## 2. Market Insight

### 2.1 行业痛点

中小餐饮老板面临的核心矛盾：**经营决策复杂度不断增加，但人力和能力没有同步增长。**

- 一家餐厅需要同时管：菜单定价、促销活动、会员运营、排班、出品效率、客户满意度、财务对账、外卖平台
- 传统 POS 只解决"收银"，不解决"经营"
- 请运营总监年薪 20-50 万，中小餐饮请不起
- 外卖平台抽成 15-25%，但老板离不开
- 数据散落在 POS、外卖平台、会员系统、Excel 里，无法形成统一视图

### 2.2 市场机会

AI 技术成熟到可以替代"运营总监"的大部分工作：

- LLM 可以分析经营数据、生成促销方案、识别会员流失
- MCP 协议让 AI 可以直接操作业务系统
- Agent 架构让每家餐厅可以拥有自己的 AI 身份
- 成本从"年薪 30 万的运营总监"降到"月费几百块的 AI 服务"

### 2.3 竞争格局

| 类型 | 代表产品 | 局限 |
|------|---------|------|
| 传统 POS | 客如云、美味不用等、二维火 | 只是工具，不做经营决策 |
| 餐饮 SaaS | 美团餐饮系统、饿了么商家版 | 平台绑定，数据不属于商户 |
| AI 客服 | 各类 chatbot | 只做对话，不能操作业务系统 |
| **FounderPOS** | — | **AI 运营合伙人 + 自主 Agent + 数据归商户** |

## 3. Target Customer

### 3.1 Primary: 一人管店型老板

- 新加坡中小餐饮（小贩中心、咖啡店、小连锁）
- 特征：老板兼厨师兼收银兼采购，没有运营团队
- 核心需求：**不是更好的工具，而是一个不休息的运营合伙人**
- 付费能力：月费 SGD 100-500

### 3.2 Secondary: 小连锁品牌

- 3-20 家门店的区域连锁
- 特征：有简单的管理层，但缺乏数据驱动能力
- 核心需求：跨门店经营分析、统一促销管理、AI 辅助决策
- 付费能力：月费 SGD 500-2000

### 3.3 Tertiary: 餐饮投资方

- 通过 RWA 机制参与餐饮经营的投资者
- 核心需求：实时经营数据透明、AI 风险评估
- 付费模式：交易抽成

## 4. Core Value Proposition

### 传统 POS vs FounderPOS

| 维度 | 传统 POS | FounderPOS |
|------|---------|------------|
| 定位 | 收银工具 | AI 运营合伙人 |
| 使用者 | 收银员操作 | 老板审批，AI 执行 |
| 菜单管理 | 人工配置 | AI 分析销量/毛利，自动建议调整 |
| 促销活动 | 人工设计 | AI 根据客流/天气/历史自动生成方案 |
| 会员运营 | 人工维护 | AI 识别流失、自动触发召回 |
| 经营分析 | 看报表 | AI 主动推送洞察和行动建议 |
| 对外交互 | 人工接电话 | Restaurant Agent 自动处理预定/包场/询价 |
| 数据归属 | 平台所有 | 商户自己的 Agent 持有 |

**一句话：给每家餐厅配一个不休息的运营总监，老板只管拍板。**

## 5. Product Architecture（四层）

```
Layer 4 ─ Agent Identity Layer
           每家餐厅 = 1 Agent + 1 Wallet
           对外开放：预定/包场/合作/询价
           Agent-to-Agent Protocol

Layer 3 ─ AI Operator Layer
           Restaurant AI Operator（内部运营大脑）
           5 个角色：菜单顾问/营销顾问/会员顾问/经营顾问/出品顾问
           模式：AI 分析 → 生成方案 → 人类审批 → 执行

Layer 2 ─ MCP Tool Layer
           每个域暴露为 MCP Tools（14 个工具）
           统一 ActionContext（actor_type / decision_source / approval）
           人类操作和 AI 操作走同一套接口

Layer 1 ─ Transaction Foundation
           POS V2 后端：Order / Catalog / CRM / Promotion / Settlement / Report
           传统 CRUD + 业务逻辑，独立可用
```

## 6. AI Operator — 5 个顾问角色

| 角色 | 感知什么 | 输出什么 |
|------|---------|---------|
| 菜单顾问 | SKU 销量、毛利、退菜率、出菜时间 | "下架3道亏损菜""推主推套餐组合" |
| 营销顾问 | 客流趋势、促销效果、竞对活动 | "周三满减草案""节日活动方案" |
| 会员顾问 | 消费频次、充值余额、流失风险 | "召回30天未到店会员""调整等级规则" |
| 经营顾问 | 日报、周报、毛利、人效、翻台率 | "本周经营摘要""异常预警" |
| 出品顾问 | 平均出菜时间、退菜原因、高峰时段 | "高峰期简化菜单建议""出品瓶颈分析" |

**运行模式：Sense → Think → Propose → Approve → Act**

- 定时巡检：每天早上生成经营摘要
- 事件驱动：出菜超时、会员流失、促销到期时触发
- 老板提问："最近生意怎么样""帮我想个促销方案"

## 7. Restaurant Agent + Wallet

每家餐厅注册后获得：

- **Agent 身份：** 名称、地址、菜系、营业时间、能力声明
- **Wallet：** 收款账户、余额、分账规则、预算控制
- **对外能力：** 接受预定、包场询价、供应商对接

### 对外交互场景

| 外部 Agent | 交互 | 举例 |
|-----------|------|------|
| 顾客 Agent | 询价/订位/点餐 | "周六晚上8人有位吗？人均预算150" |
| 企业 Agent | 包场/团建/定制 | "30人团建，预算5000，有方案吗" |
| 供应商 Agent | 报价/采购 | "明天需要50斤牛肉，给个价" |
| 平台 Agent | 上架/促销/结算 | 美团/饿了么 Agent 自动对接 |

### 风险分级

- LOW → AI 自动回复（查营业时间、菜单、基本报价）
- MEDIUM → AI 生成方案 → 老板审批（包场报价、特殊需求）
- HIGH → 转人工（大额交易、合作协议）

## 8. Restaurant Agent Network（P4）

当足够多餐厅加入后，形成网络效应：

- **集体智慧：** 1000 家餐厅的匿名数据，让每家店的 AI 更聪明
- **经营基准：** "同区域同菜系餐厅毛利中位数 62%，你 55%"
- **趋势预警：** "你这区域本周外卖普遍下降 15%"
- **联合采购：** 多家店一起向供应商 Agent 压价
- **互相导流：** "我们满了，推荐隔壁同菜系"

## 9. RWA（Real World Asset）机制

每家餐厅可基于真实经营数据发行链上资产：

- **数据来源：** AI Operator 持续产出的经营分析、营收数据、风险评级
- **发债条件：** 连续 N 个月经营数据良好、AI 风险评分达标
- **投资者视角：** 实时看到 AI 生成的经营报告，不是人工编的
- **智能合约：** 基于 Agent Wallet 的收入自动分账还款

## 13. Risk

| 风险 | 应对 |
|------|------|
| AI 建议质量差 → 老板不信任 | 严格风险分级，HIGH 必须人工；持续优化 prompt |
| 数据隐私 | 匿名化处理，opt-in 模式，数据归商户 |
| Agent 被攻击 | 对外 Agent 有预算上限、操作白名单 |
| RWA 合规 | 分区域合规评估，先新加坡 MAS 框架 |
