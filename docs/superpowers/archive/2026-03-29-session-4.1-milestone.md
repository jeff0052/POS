## 设计 Lessons

### L1: ticketNo 用 DB PK，不用时间戳

**问题：** 初版用 `"KT-{storeId}-{yyyyMMddHHmmss}-{seq}"`，同一秒内两单提交会碰撞。改用毫秒仍不安全（并发）。

**正确做法：** `"KT-{storeId}-{submittedOrderId}-{seq}"`。`submittedOrderId` 是 DB 自增 PK，全局唯一，不依赖时间。

**原则：** 需要唯一标识符时，优先用数据库保证的唯一性（PK / UUID），不要自己造时间戳组合键。

---

### L2: 跨租户引用必须显式验证 storeId

**问题：** `sku.stationId` 是裸 ID，没有验证 station 是否属于当前 order 的 store。SKU 如果被错误配置绑到另一个商家的 station，会静默路由到错误租户。

**正确做法：** 加载 station 后立即检查：
```java
if (!station.getStoreId().equals(order.getStoreId())) {
    throw new IllegalStateException("Station " + stationId
        + " belongs to store " + station.getStoreId()
        + ", not order store " + order.getStoreId());
}
```

**原则：** 任何跨模块的 ID 引用（SKU→Station、Order→Table 等），加载后必须验证 storeId 归属，不能假设数据一致性。

---

### L3: option_snapshot_json 全管道思维

**问题：** DDL 列（V004/V012）存在已久，但 Java 实体从未映射，更没有在请求模型→命令→实体→厨房票→merge key 全链路传递。分四轮才完整修完：

1. 实体映射 + `toSubmittedItem()` 传递
2. 请求模型 + 命令 record 缺字段 → 全部补上
3. `mergeQrItems()` clone 没复制 `optionSnapshotJson`
4. `mergeQrItems()` match key 只用 `skuId + remark`，不同 option 的同款 SKU 被合并

**最终 match key：**
```java
item.getSkuId().equals(incoming.skuId())
&& Objects.equals(item.getItemRemark(), incoming.remark())
&& Objects.equals(item.getOptionSnapshotJson(), incoming.optionSnapshotJson())
```

**原则：** 新增任何数据字段，必须追踪完整 pipeline：
`DDL → Entity → Command → Request → Controller → Service 处理 → 子实体 → clone 逻辑 → match/dedup key`
任何一段断开都是 silent data loss。

---

### L4: YAGNI 边界要写进代码注释

**问题：** 审查者提出 ticket cancel 应同步作废 billable order items（P1-3）。这是合理的业务需求，但超出 Session 4.1 范围——`kitchen_ticket_items` 没有 `submitted_order_item_id` FK，`submitted_order_items` 没有 `item_status` 列。

**处理：** 成功推回，并在代码中留明确 TODO：
```java
// TODO Session 4.2: item-level cancellation
// Requires: submitted_order_items.item_status column
//           kitchen_ticket_items.submitted_order_item_id FK
```

**原则：** YAGNI 推回时必须有 spec/DDL 依据，不能只说"不在范围"。推回成功后用 TODO 注释标记跨 session 遗留，防止被后来者误以为是遗漏。

---

### L5: 开发环境 PATH 问题会静默阻断 CI

**问题：** `.zshrc` 没配 `JAVA_HOME` 和 Maven PATH，导致整个 session 无法运行 `mvn test`。只能用历史 surefire-reports 作为测试通过的间接证据。

**修复：** 在 `.zshrc` 里显式配置 `JAVA_HOME` 和 `mvn` PATH，新开终端即生效。

**原则：** 项目初始化时验证 `java -version` 和 `mvn -version` 可用。CI 脚本用绝对路径或在脚本顶部 `source ~/.zshrc`。

---

### L6: StoreAccessEnforcer 模式

`StoreAccessEnforcer` 是本 session 引入的跨模块共享组件，解决 Phase 3 遗留问题：各 Service 里重复 `enforceStoreAccess()` 逻辑。

**设计要点：**
- SUPER_ADMIN 绕过 merchantId 检查（null/0 merchant 的 platformAdmin 场景）
- 权限检查分离：`enforce(storeId)` vs `enforcePermission(permissionCode)`
- 放在 `v2/common/application/`，任何模块都可注入

**原则：** 安全类横切逻辑（访问控制、租户隔离）应提取为共享组件，不要在每个 Service 里各写一遍。

**代码位置：** `TicketStatusService.java` 中的 `// TODO Session 4.2` 注释

**优先级：** P1（影响结账准确性）

**修复步骤：**
1. `source ~/.zshrc`（已修复 PATH 问题）
2. 在 `pos-backend/` 目录执行 `./mvnw test`
3. 如果 ByteBuddy 报错，在 `pom.xml` surefire plugin 加：
   ```xml
   <argLine>-javaagent:${settings.localRepository}/org/mockito/mockito-core/.../mockito-core-x.x.x.jar</argLine>
   ```
   或升级到 Mockito 5.x（内置 inline mock）

**优先级：** P1（需要在下一个 session 开工前确认测试绿）
