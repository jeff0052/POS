## Section 1: Shared Invariants + Data Layer

### 1A. 状态机

**settlement_payment_holds.hold_status:**
```
HELD → CONFIRMED  (支付成功, confirmStacking)
HELD → RELEASED   (支付失败/放弃/超时, releaseStacking)
```

**member_coupons.coupon_status:**
```
AVAILABLE → LOCKED  (collectStacking 锁券, CAS)
LOCKED → USED       (confirmStacking)
LOCKED → AVAILABLE  (releaseStacking / 10min 超时回收)
```

**payment_attempts.attempt_status (现有 + 新增):**
```
现有: PENDING_CUSTOMER → SUCCEEDED → SETTLED  (P2-A: confirmStacking 推进 SUCCEEDED→SETTLED)
      PENDING_CUSTOMER → FAILED
      PENDING_CUSTOMER → EXPIRED
新增: FAILED          → REPLACED  (switchMethod 标记旧 attempt)
      PENDING_CUSTOMER → REPLACED  (P1-B: releaseStacking 强制取消 live attempt)
```

**settlement_records.final_status (新增):**
```
现有: SETTLED
新增: PENDING → SETTLED  (confirmStacking)
      PENDING → CANCELLED (releaseStacking)
```

**锁 owner:** 统一用 `table_sessions.id` (BIGINT)，用于：
- `settlement_payment_holds.table_session_id`
- `member_coupons.locked_by_session`

超时回收和定向释放都按此 key 操作。

**hold_type canonical 值:** `POINTS | COUPON | CASH_BALANCE | EXTERNAL`

**锁顺序规则（防死锁）：** 所有事务必须按以下顺序获取行锁：
1. `settlement_records` FOR UPDATE（先锁结算记录）
2. `table_sessions` FOR UPDATE（后锁 session，仅 finalize 时需要）
3. `member_accounts` 条件更新（原子操作，不持锁）
4. `member_coupons` CAS（原子操作，不持锁）

collectStacking 先对 masterSession 做 `FOR UPDATE`（mutex，防并发重复创建），锁内再做幂等检查。confirmStacking / releaseStacking / switchMethod 先锁 settlement_records，finalize 时再锁 session。两条路径锁顺序不交叉，无死锁风险。

## Section 2: PaymentStackingService

### 四阶段

**a. previewStacking(storeId, tableId)**
- rejectMergedChildSession → buildSessionChain → 聚合 unpaid submitted orders
- 取 memberId：收集 session chain 内所有 non-null memberId，**`distinct count > 1` → 抛异常拒绝混合会员结算**。0 个 → 无会员路径。1 个 → 查余额和可用券。
- 干跑叠加顺序：积分抵扣 → 券减免 → 储值扣减 → 剩余走外部支付
- 返回 `StackingPreviewDto`：每步的 holdType + amount + 可选券列表 + 外部支付剩余
- **不冻结，不写 DB**

**b. collectStacking(storeId, tableId, stackingChoices)**
- stackingChoices: `{ usePoints: boolean, couponId: Long?, useCashBalance: boolean, externalPaymentMethod: String }`
- rejectMergedChildSession + buildSessionChain + memberId 校验（distinct ≤ 1）
- **并发互斥（防重复创建）：** 先对 masterSession 做 `SELECT ... FOR UPDATE`，锁住 session 行作为 mutex。两个并发 collect 都会争同一行锁，只有一个先进，另一个阻塞等待。
- **幂等检查（在锁内执行）：** 获得 session 锁后，查 `settlementRecordRepository.findByStackingSessionIdAndFinalStatus(masterSession.getId(), "PENDING")` → 有 → 返回已有结果（后进的那个请求走这里）。

  > **锁顺序说明：** collectStacking 先锁 session（无 settlement 可锁）。confirm/release/switch 先锁 settlement（已创建）、后锁 session（仅 finalize 时）。两条路径无交叉，不会死锁。
- 重新计算各步金额（防 preview 和 collect 之间数据变化）
- **Step 1: 创建 settlement_record**
  - `finalStatus = "PENDING"`
  - `active_order_id = "STACK-" + UUID`（全局唯一，不撞现有唯一约束）
  - `stacking_session_id = masterSession.getId()`
  - `cashierId = AuthContext.current().userId()`（从安全上下文取，可 null）

  > **注:** `STACK-<uuid>` 只是 persistence key，不对外暴露。外部系统和 API 一律用 settlementId / settlementNo。

- **Step 2: 按顺序冻结，每步写 hold（带 settlement_record_id）**
  1. **POINTS:** 条件更新 `UPDATE member_accounts SET frozen_points = frozen_points + :x WHERE member_id = :id AND (points_balance - frozen_points) >= :x`，affected_rows = 0 → 余额不足拒绝 + 回滚。写 hold(step_order=1)。
  2. **COUPON:** 调 CouponLockingService.lockCoupon(couponId, expectedVersion, sessionId)。写 hold(step_order=2)。
  3. **CASH_BALANCE:** 条件更新同积分模式。写 hold(step_order=3)。
  4. **EXTERNAL:** 写 hold(step_order=4, hold_amount_cents = payable - 前三步之和)。如果 > 0，创建 payment_attempt（带 settlement_record_id）。
- **Step 3: 事务提交**
- **Step 4: 事务提交后**
  - 如果有 payment_attempt → 调 VibeCash API 创建支付链接
    - **确定失败（4xx/业务拒绝）→ releaseStacking(settlementId)**
    - **未知（timeout/5xx/网络异常）→ 保留 PENDING**，由 retry/switch-method/过期回收处理
  - 如果 external = 0 → 直接调 confirmStacking(settlementId)
- 返回 `StackingCollectResultDto`：settlementId + holdIds + externalPaymentUrl（如有）

**c. confirmStacking(settlementId)**
- `findByIdForUpdate(settlementId)` — 行锁
- **幂等：** 已 SETTLED → 直接返回。已 CANCELLED → 抛异常。
- **只允许 PENDING → SETTLED**
- 如果有 EXTERNAL hold → 校验关联 `payment_attempt.attempt_status = 'SUCCEEDED'`，否则拒绝
- holds HELD → CONFIRMED + confirmedAt
- `points_balance -= X, frozen_points -= X`
- `cash_balance_cents -= X, frozen_cash_cents -= X`
- 调 CouponLockingService.confirmCoupon(couponId, sessionId, usedOrderId, usedStoreId)
- **[P2-A] 将关联 payment_attempt 状态推进 SUCCEEDED → SETTLED**（`UPDATE payment_attempts SET attempt_status='SETTLED' WHERE id=:id AND attempt_status='SUCCEEDED'`；幂等，已 SETTLED 无操作）
- settlement_record: `finalStatus = 'SETTLED'` + 填充叠加明细字段（pointsDeductCents, pointsDeducted, cashBalanceDeductCents, couponDiscountCents, externalPaymentCents, couponId）
- 调 `finalizeTableSettlement(storeId, tableId, masterSession, mergedSessions)`：submitted_orders → SETTLED, sessions → CLOSED, tables → PENDING_CLEAN, 删 active orders

**d. releaseStacking(settlementId, reason)**
- `findByIdForUpdate(settlementId)` — 行锁
- **幂等：** 已 CANCELLED → 直接返回。已 SETTLED → 抛异常。
- **只允许 PENDING → CANCELLED**
- **[P1-B] 先处理 live payment attempts（同一事务内）：**
  - 查 `payment_attempts WHERE settlement_record_id = :id AND attempt_status IN ('PENDING_CUSTOMER', 'SUCCEEDED')`
  - **若存在 `SUCCEEDED` attempt → 立即抛 `CannotReleaseAlreadyChargedSettlementException`，中止 release，事务回滚**。settlement 保持 PENDING，等待 confirmStacking 到来。调用方（scheduler/API 层）需捕获此异常，记 `CRITICAL` 日志并人工介入。
  - 对每个 `PENDING_CUSTOMER` attempt：CAS 更新 `SET attempt_status='REPLACED' WHERE attempt_status='PENDING_CUSTOMER'`
  - 若 CAS 失败（affected_rows=0）→ attempt 在查询和更新之间已变 SUCCEEDED（极窄竞态）→ 回滚并抛同一 `CannotReleaseAlreadyChargedSettlementException`
- holds HELD → RELEASED + releasedAt + releaseReason
- `frozen_points -= X, frozen_cash_cents -= X`（条件更新，防下溢）
- 调 CouponLockingService.releaseCoupon(couponId, sessionId)
- settlement_record: `finalStatus = 'CANCELLED'`

### VibeCash webhook 适配

```
handleWebhook:
  1. 查 payment_attempt by providerPaymentId
  2. 如果 attempt.attemptStatus = 'REPLACED' → 只记日志，不触发 confirm/release
  3. 如果 attempt.attemptStatus = 'PENDING_CUSTOMER':
     a. payment.succeeded → attempt SUCCEEDED
        → 如果 attempt.settlementRecordId != null → confirmStacking(settlementId)
        → 否则 → 走旧路径 collectForTable
     b. payment.failed → attempt FAILED
        → **[P1-A] 保留 settlement PENDING，不触发 releaseStacking**
        → 只记日志："payment failed, settlement remains PENDING, switch-method window open"
        → switchMethod 可在此后发起新 attempt；前端应提示用户重试或换付款方式
     c. checkout.session.expired → attempt EXPIRED
        → 如果 attempt.settlementRecordId != null → releaseStacking(settlementId, "CHECKOUT_EXPIRED")
  4. 其他状态 → 幂等忽略
```

> **[P1-A] payment.failed 和 checkout.session.expired 的区别：**
> - `payment.failed`：支付失败，用户可以重试（换卡/换码），对应 switchMethod 窗口。Settlement 保持 PENDING，持有冻结资产。
> - `checkout.session.expired`：支付链接过期，无法继续使用当前 URL。Settlement 释放冻结，前端需重新发起 collectStacking（保留会员 holds 语义不变）。
> - **如果前端希望"失败即放弃"：** 应显式调用 `release-stacking` API，而非依赖 webhook 自动释放。

> **重要：** 现有 `VibeCashPaymentApplicationService.startPayment()` 禁止设 `settlementRecordId`。webhook 路由靠 `settlementRecordId != null` 区分旧路径和新 stacking 路径。如果 startPayment 意外设了此字段，会导致非 stacking 支付走 confirmStacking 逻辑。

### finalizeTableSettlement 提取

提取到新的 `TableSettlementFinalizer` service（`v2/settlement/application/service/`），供 `CashierSettlementApplicationService.collectForTable()`（旧路径）和 `PaymentStackingService.confirmStacking()`（新路径）共用：
- submitted_orders → SETTLED + settledAt
- master session → CLOSED + closedAt
- merged sessions → CLOSED + closedAt + mergedIntoSessionId = null
- merged tables → PENDING_CLEAN
- master table → PENDING_CLEAN
- 删 active orders

## Section 3: CouponLockingService

### 方法

**a. lockCoupon(couponId, expectedVersion, sessionId)**
- CAS 更新：
  ```sql
  UPDATE member_coupons
  SET coupon_status = 'LOCKED', lock_version = lock_version + 1,
      locked_by_session = :sessionId, locked_at = NOW()
  WHERE id = :couponId AND coupon_status = 'AVAILABLE' AND lock_version = :expectedVersion
  ```
- affected_rows = 0 → 抛 `CouponAlreadyLockedException`
- 返回更新后的 lock_version

**b. releaseCoupon(couponId, sessionId)**
- ```sql
  UPDATE member_coupons
  SET coupon_status = 'AVAILABLE', locked_by_session = NULL, locked_at = NULL,
      lock_version = lock_version + 1
  WHERE id = :couponId AND coupon_status = 'LOCKED' AND locked_by_session = :sessionId
  ```
- 幂等：已 AVAILABLE → 无操作

**c. confirmCoupon(couponId, sessionId, usedOrderId, usedStoreId)**
- ```sql
  UPDATE member_coupons
  SET coupon_status = 'USED', used_at = NOW(), used_order_id = :usedOrderId,
      used_store_id = :usedStoreId
  WHERE id = :couponId AND coupon_status = 'LOCKED' AND locked_by_session = :sessionId
  ```
- 幂等：已 USED 且 `used_order_id = :usedOrderId` → 无操作。已 USED 且 `used_order_id != :usedOrderId` → 抛冲突异常。

**d. reclaimExpiredLocks()** — 定时任务每分钟调用
- 查 `SELECT * FROM member_coupons WHERE coupon_status = 'LOCKED' AND locked_at < NOW() - INTERVAL 10 MINUTE`
- **先按 `locked_by_session` / `settlement_record_id` 去重**，避免同一 settlement 因多张过期券被重复触发
- 对每个去重后的 settlement：
  - 查 `settlement_payment_holds WHERE coupon_id = :couponId AND hold_type = 'COUPON' AND hold_status = 'HELD'` → 取 settlement_record_id
  - 如果有关联 settlement → `releaseStacking(settlementId, reason='COUPON_LOCK_EXPIRED')`（由 releaseStacking 原子释放所有 holds）
  - 如果无关联 settlement（孤儿锁）→ 单张 `releaseCoupon(couponId, sessionId)`

### 设计决策

1. CAS 不用 SELECT FOR UPDATE——乐观锁比悲观锁轻，并发锁同一券的概率低
2. **券超时直接触发整个 settlement release**——不做部分释放，因为金额分解已经变了
3. 10 分钟超时硬编码，不做 merchant_configs 可配置（YAGNI）

## Section 4: PaymentRetryService

### 方法

**switchMethod(storeId, tableId, paymentAttemptId, newPaymentScheme)**

```
1. 查 attempt by paymentAttemptId（业务键，非 DB id）
2. 校验 attempt.storeId == route.storeId AND attempt.tableId == route.tableId
3. findByIdForUpdate(attempt.settlementRecordId) — 行锁，和 confirm/release 共用
4. 在锁内重验 settlement.finalStatus == PENDING，否则拒绝
5. 校验 attempt.attemptStatus == 'FAILED'（EXPIRED/REPLACED 不允许 switch）
6. 校验 retry_count < max_retries
7. CAS 更新旧 attempt：
   UPDATE SET attempt_status='REPLACED'
   WHERE payment_attempt_id=:id AND attempt_status='FAILED' AND replaced_by_attempt_id IS NULL
   → affected_rows=0 → 抛 ConcurrentModificationException
8. 创建新 attempt：
   - 复制 storeId/tableId/tableSessionId/sessionRef/settlementAmountCents/currencyCode
   - settlement_record_id = 同一个 PENDING settlement
   - parent_attempt_id = 旧 attempt.id
   - retry_count = 旧.retry_count + 1
   - payment_scheme = newPaymentScheme
   - attempt_status = PENDING_CUSTOMER
9. 回填旧 attempt replaced_by_attempt_id = 新 attempt.id
10. 事务提交
11. 事务后调 VibeCash API（分级处理：确定失败→新 attempt FAILED / 未知→保留 PENDING_CUSTOMER）
```

返回 `PaymentRetryResultDto`：newAttemptId + checkoutUrl

### API

`POST /api/v2/stores/{storeId}/tables/{tableId}/payment/switch-method`
```json
Request:  { paymentAttemptId: "pa_xxx", newPaymentScheme: "ALIPAY_QR" }
Response: { newAttemptId: "pa_yyy", checkoutUrl: "https://..." }
```

权限：`PAYMENT_SWITCH`

SecurityConfig 加：
```java
.requestMatchers(HttpMethod.POST, "/api/v2/stores/*/tables/*/payment/switch-method")
    .hasAuthority("PAYMENT_SWITCH")
```

### 设计决策

1. 不做自动重试——用户主动选新支付方式
2. 复用同一个 PENDING settlement——switch 不释放已冻结的积分/储值/券
3. retry_count 在 attempt 链上递增——max_retries 默认 3
4. EXPIRED attempt 不允许 switch——应走 releaseStacking 重来

## Section 5: 定时任务 + API 汇总

### 定时任务

| 任务 | 频率 | 逻辑 |
|------|------|------|
| 券超时回收 | 每分钟 | CouponLockingService.reclaimExpiredLocks() |
| **[P1-C] PENDING 结算超时回收** | 每分钟 | PaymentStackingService.reclaimPendingSettlements() |

**reclaimPendingSettlements()** — 定时任务每分钟调用
- 查 `SELECT * FROM settlement_records WHERE final_status = 'PENDING' AND created_at < NOW() - INTERVAL 30 MINUTE`
- **注：** 30 分钟硬编码（VibeCash checkout 链接最长有效期上限 + 缓冲）；不做 merchant_configs 可配置（YAGNI）
- 对每条 settlement：
  - 查关联 `payment_attempts WHERE settlement_record_id = :id`，取最新 attempt
  - **若最新 attempt 状态为 `SUCCEEDED`** → 说明 webhook 已将 attempt 推进 SUCCEEDED，但 `confirmStacking` 事务失败，且不保证 webhook 重发。**直接调 `confirmStacking(settlementId)`**（幂等，已 SETTLED 直接返回；`confirmStacking` 内部会校验 attempt.status = SUCCEEDED 才允许确认，因此是安全调用）。若 confirm 失败（事务异常）→ 记 CRITICAL 日志，等下一分钟重试。
  - **若最新 attempt 状态为 `PENDING_CUSTOMER` 且 `created_at > NOW() - INTERVAL 30 MINUTE`**（年轻，链接可能仍有效）→ 跳过（正常支付中）
  - **若最新 attempt 状态为 `PENDING_CUSTOMER` 且 `created_at < NOW() - INTERVAL 30 MINUTE`**（老于 30 分钟，provider 侧链接已必然过期）→ 调 `releaseStacking(settlementId, "ATTEMPT_EXPIRED_NO_WEBHOOK")`
    - P1-B 的 CAS 会把这个 PENDING_CUSTOMER attempt 标为 REPLACED
    - 若 VibeCash 后续仍补发 webhook（极端延迟）→ attempt 已 REPLACED，webhook 静默忽略
  - **若无 attempt，或最新 attempt 状态为 `FAILED`/`EXPIRED`/`REPLACED`** → 调 `releaseStacking(settlementId, "SETTLEMENT_TIMEOUT")`
  - `releaseStacking` 抛 `CannotReleaseAlreadyChargedSettlementException` → 记 CRITICAL，跳过该条
- **设计决策：** 区分"链接还年轻"和"链接必然过期"——前者跳过，后者主动回收。避免了 expired webhook 丢失导致冻结永久卡住的问题，同时不误 cancel 正在进行中的支付。

### API 端点

| 方法 | 路径 | 权限 | 说明 |
|------|------|------|------|
| POST | `/api/v2/stores/{storeId}/tables/{tableId}/payment/preview-stacking` | SETTLEMENT_STACKING | 预览叠加明细 |
| POST | `/api/v2/stores/{storeId}/tables/{tableId}/payment/collect-stacking` | SETTLEMENT_STACKING | 冻结 + 发起支付 |
| POST | `/api/v2/stores/{storeId}/tables/{tableId}/payment/{settlementId}/confirm-stacking` | SETTLEMENT_STACKING | 确认结算（通常由 webhook 触发） |
| POST | `/api/v2/stores/{storeId}/tables/{tableId}/payment/{settlementId}/release-stacking` | SETTLEMENT_STACKING | 释放冻结 |
| POST | `/api/v2/stores/{storeId}/tables/{tableId}/payment/switch-method` | PAYMENT_SWITCH | 换支付方式 |

### SecurityConfig 重构

**移除现有 `/api/v2/stores/*/tables/*/payment/**` → `SETTLEMENT_COLLECT` 通配符。** 改为显式列出每个 payment 端点，避免通配符吞掉细粒度权限。

```java
// Payment endpoints — explicit matchers, NO wildcard (BEFORE /stores/** permitAll)
// Existing table payment endpoints
.requestMatchers(HttpMethod.GET, "/api/v2/stores/*/tables/*/payment/preview").hasAuthority("SETTLEMENT_COLLECT")
.requestMatchers(HttpMethod.POST, "/api/v2/stores/*/tables/*/payment").hasAuthority("SETTLEMENT_COLLECT")
.requestMatchers(HttpMethod.POST, "/api/v2/stores/*/tables/*/payment/collect").hasAuthority("SETTLEMENT_COLLECT")
.requestMatchers(HttpMethod.POST, "/api/v2/stores/*/tables/*/payment/vibecash").hasAuthority("SETTLEMENT_COLLECT")
.requestMatchers(HttpMethod.GET, "/api/v2/stores/*/tables/*/payment/attempts/*").hasAuthority("SETTLEMENT_COLLECT")
// New stacking endpoints
.requestMatchers(HttpMethod.POST, "/api/v2/stores/*/tables/*/payment/preview-stacking").hasAuthority("SETTLEMENT_STACKING")
.requestMatchers(HttpMethod.POST, "/api/v2/stores/*/tables/*/payment/collect-stacking").hasAuthority("SETTLEMENT_STACKING")
.requestMatchers(HttpMethod.POST, "/api/v2/stores/*/tables/*/payment/*/confirm-stacking").hasAuthority("SETTLEMENT_STACKING")
.requestMatchers(HttpMethod.POST, "/api/v2/stores/*/tables/*/payment/*/release-stacking").hasAuthority("SETTLEMENT_STACKING")
