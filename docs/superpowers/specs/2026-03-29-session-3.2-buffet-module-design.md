# Session 3.2 — Buffet Module Design Spec

**Date:** 2026-03-29
**Status:** DRAFT

## 1. Goal

Implement the full buffet lifecycle: package CRUD, table start, timed session, buffet-specific menu, per-person limit enforcement, and checkout pricing (head fee + surcharge + extra + overtime).

## 2. Scope

**In scope:**
- BuffetPackageEntity + BuffetPackageItemEntity (new)
- TableSessionEntity buffet field mapping (existing DDL, new Java fields)
- 4 services: PackageCRUD, Session, Menu, Pricing
- 1 validation method: per-person limit check
- 3 controllers, ~10 endpoints
- Unit tests for all services

**Out of scope:**
- Modifying legacy V1 order services (QrMenuApplicationService, ActiveTableOrderApplicationService)
- Settlement integration (Session 4.x)
- Merge-table buffet logic (J11 A4 — future session)

## 4. Service Design

### 4.1 BuffetPackageService

**Package:** `v2.catalog.application.service`

Manages buffet package CRUD and SKU bindings.

**Methods:**
- `createPackage(storeId, CreateBuffetPackageRequest)` → BuffetPackageDto
- `updatePackage(storeId, packageId, UpdateBuffetPackageRequest)` → BuffetPackageDto
- `deletePackage(storeId, packageId)` → void (soft delete → INACTIVE)
- `listPackages(storeId)` → List<BuffetPackageDto>
- `getPackage(storeId, packageId)` → BuffetPackageDto
- `bindSku(storeId, packageId, BindSkuRequest)` → BuffetPackageItemDto
- `updatePackageItem(storeId, packageId, itemId, UpdateBindingRequest)` → BuffetPackageItemDto
- `unbindSku(storeId, packageId, itemId)` → void
- `listPackageItems(storeId, packageId)` → List<BuffetPackageItemDto>

**Permissions:**
- Write: `BUFFET_MANAGE` + store-scope access
- Read: `MENU_VIEW || BUFFET_MANAGE` + store-scope access

**API contract deviation:** 85-api-contract.md §4 uses `MENU_MANAGE` for buffet package write operations. This spec uses `BUFFET_MANAGE` instead, which is a dedicated permission in the RBAC seed (V102 line 54). Rationale: RBAC seed intentionally separates catalog management (`MENU_MANAGE`) from buffet management (`BUFFET_MANAGE`); CASHIER role has `BUFFET_START` but not `MENU_MANAGE`, while STORE_MANAGER has both. The API contract should be updated to match.

### 4.2 BuffetSessionService

**Package:** `v2.catalog.application.service`

Manages buffet lifecycle on a table session.

**Methods:**
- `startBuffet(storeId, tableId, StartBuffetRequest{packageId, guestCount, childCount})` → BuffetStatusDto
- `getBuffetStatus(storeId, tableId)` → BuffetStatusDto
- `validateBuffetOrder(sessionId, List<OrderItemRequest>)` → BuffetOrderValidationResult

**startBuffet logic:**
1. Permission: `BUFFET_START` + store-scope access
2. Load package → verify store ownership + ACTIVE status
3. Find open session for table → verify not already BUFFET (409 if so)
4. Set: dining_mode=BUFFET, buffet_package_id, buffet_started_at=NOW(), buffet_ends_at=NOW()+duration, buffet_status=ACTIVE, guest/child count
5. Return status DTO

**getBuffetStatus logic:**
1. No permission required (frontend polling)
2. Find session by storeId+tableId where status=OPEN
3. Real-time compute from timestamps:
   - remainingMinutes = buffet_ends_at - NOW()
   - if remaining > warningBefore → ACTIVE
   - if 0 < remaining <= warningBefore → WARNING
   - if remaining <= 0 → OVERTIME, compute overtime fee
4. Return DTO (does NOT write DB — snapshot only)

**validateBuffetOrder logic:**
1. Load session → get buffet_package_id
2. For each item:
   - Find packageItem by (packageId, skuId)
   - If not found → mark as extra (isBuffetIncluded=false, surchargeCents=0)
   - If found + EXCLUDED → reject with 400
   - If found + maxQtyPerPerson set → check alreadyOrdered + quantity <= limit × guestCount
   - Annotate: isBuffetIncluded, buffetSurchargeCents, buffetPackageId
3. Return validation result list

### 4.3 BuffetMenuService

**Package:** `v2.catalog.application.service`

Serves the customer-facing buffet menu filtered by package.

**Methods:**
- `getBuffetMenu(storeId, packageId)` → List<MenuCategoryDto>

**Logic:**
1. No permission required (customer endpoint)
2. Load all package items WHERE inclusionType != EXCLUDED, JOIN skus → products → categories
3. Batch-resolve image URLs via ImageUploadService (CDN pattern from Session 3.1)
4. Group by category, annotate each SKU with: inclusionType, surchargeCents, maxQtyPerPerson
5. Return using extended MenuSkuDto

### 4.4 BuffetPricingService

**Package:** `v2.catalog.application.service`

Computes buffet bill at checkout time.

**Methods:**
- `calculateBuffetTotal(storeId, sessionId)` → BuffetBillDto

**Pricing formula:**
```
headFee     = package.priceCents × guestCount
            + (package.childPriceCents ?? 0) × childCount
surcharge   = SUM(items WHERE isBuffetIncluded=true AND buffetSurchargeCents > 0
                   → quantity × buffetSurchargeCents)
extra       = SUM(items WHERE isBuffetIncluded=false → lineTotalCents)
overtime    = max(0, actualMinutes - durationMinutes - graceMinutes)
              capped at maxOvertimeMinutes
              × overtimeFeePerMinuteCents
grandTotal  = headFee + surcharge + extra + overtime
```

**Query path:** session (by sessionId) → find all submitted_orders (by table_session_id) → aggregate submitted_order_items with buffet fields. `storeId` is used for store-scope permission check, not for the query itself.

**Permission:** `BUFFET_START` (cashier-level)

### 4.5 ENDED state transition

`buffet_status=ENDED` is written when settlement completes for a BUFFET session. This is NOT handled by Session 3.2 (settlement is Session 4.x). The contract for settlement integration:

1. Settlement service calls `BuffetPricingService.calculateBuffetTotal(storeId, sessionId)` to get the buffet bill breakdown
2. After successful payment, settlement service updates: `session.buffet_status=ENDED`, `session.buffet_overtime_minutes=actualOvertimeMinutes`
3. If overtime exceeds `max_overtime_minutes`, `getBuffetStatus` returns a `forceCheckout=true` flag for frontend notification — but does NOT auto-close the session

### 4.6 Order validation wiring

`validateBuffetOrder()` is an advisory method in Session 3.2. It is not wired into any order submission flow because this session does not modify legacy order services. The expected integration point is the future new order service (Phase 2+), which will call `validateBuffetOrder()` before persisting order items. Until then, buffet field annotation and limit enforcement are testable via unit tests but not enforced in production order flow.

### 5.3 Unified menu endpoint integration (J02 step 4)

J02 specifies the customer fetches buffet menu via the unified endpoint:
```
GET /api/v2/stores/{storeId}/menu?diningMode=BUFFET&packageId={pkgId}
```

The existing `MenuQueryService.queryMenu()` already accepts `diningMode`. When `diningMode=BUFFET` and `packageId` is present, it delegates to `BuffetMenuService.getBuffetMenu(storeId, packageId)` instead of the normal product→sku path. This keeps the QR ordering frontend unchanged — it always calls `/menu` with the appropriate diningMode.

No separate `/buffet-menu` controller is needed. The `MenuV2Controller` adds `packageId` as an optional query parameter.

## 6. DTOs

**BuffetPackageDto:** packageId, packageCode, packageName, description, priceCents, childPriceCents, childAgeMax, durationMinutes, warningBeforeMinutes, overtimeFeePerMinuteCents, overtimeGraceMinutes, maxOvertimeMinutes, packageStatus, applicableTimeSlots, applicableDays, sortOrder, imageUrl

**BuffetPackageItemDto:** itemId, skuId, skuCode, skuName, inclusionType, surchargeCents, maxQtyPerPerson, sortOrder

**BuffetStatusDto:** sessionId, buffetStatus, startedAt, endsAt, remainingMinutes, overtimeMinutes, overtimeFeeCents, guestCount, childCount, packageName, packagePriceCents, forceCheckout(boolean, true when overtime > maxOvertimeMinutes)

**BuffetBillDto:** guestCount, childCount, headFeeCents, surchargeTotal, extraTotal, overtimeMinutes, overtimeFeeCents, grandTotal, packageName, packagePriceCents, childPriceCents, durationMinutes, overtimeGraceMinutes, maxOvertimeMinutes

**BuffetMenuSkuDto:** extends MenuSkuDto fields + buffetInclusionType, buffetSurchargeCents, maxQtyPerPerson

**BuffetOrderValidationResult:** List<ValidatedItem> { skuId, isBuffetIncluded, buffetSurchargeCents, buffetPackageId, rejected, rejectReason }

**CreateBuffetPackageRequest / UpdateBuffetPackageRequest:** mirror writable fields from BuffetPackageDto, using `imageId` (not `imageUrl`) for writes

**StartBuffetRequest:** packageId, guestCount, childCount

**BindSkuRequest:** skuId, inclusionType, surchargeCents, maxQtyPerPerson

**UpdateBindingRequest:** inclusionType, surchargeCents, maxQtyPerPerson (same as BindSkuRequest minus skuId — SKU identity is immutable after binding)

## 7. Error Handling

| Scenario | HTTP | Message |
|---|---|---|
| Package not found in store | 404 | "Buffet package not found in this store" |
| Package not ACTIVE | 400 | "Buffet package is not active" |
| Duplicate buffet start | 409 | "Buffet already started for this table" |
| Per-person limit exceeded | 400 | "Item {sku} limited to {n} per person, {m} already ordered" |
| EXCLUDED item ordered | 400 | "Item {sku} is excluded from this buffet package" |
| No open session | 404 | "No open session for this table" |
| Session not BUFFET mode | 400 | "Table session is not in buffet mode" |
| Duplicate SKU binding | 409 | "SKU already bound to this package" |
| Permission denied | 403 | Reuse existing pattern |
| Store access denied | 403 | Reuse existing pattern |

## 10. Design Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Module location | v2/catalog | Avoids new module; buffet packages are catalog config. Session access via repository interface. |
| Schema source | V082/V083 full version | J02 requires three-state inclusion_type + max_qty_per_person |
| buffet_status in DB | Snapshot, not real-time | Avoids scheduled jobs. getBuffetStatus computes from timestamps. DB written only at session end. |
| BuffetMenu vs MenuQueryService | Delegate from unified /menu | J02 step 4 uses `/menu?diningMode=BUFFET&packageId=X`. MenuQueryService delegates to BuffetMenuService when BUFFET+packageId. |
| Legacy service modification | None | New system only. validateBuffetOrder is advisory until new order service. |
| No new migrations | Java mapping only | All DDL exists via V066/V067/V074 catch-up. V082/V083 from final-exec-spec is authoritative (not doc/75). |
| Permission source | BUFFET_MANAGE (not MENU_MANAGE) | RBAC seed V102 has dedicated BUFFET_MANAGE. API contract §4 says MENU_MANAGE — deviation documented, contract should be updated. |
| ENDED state | Written by settlement | Session 3.2 does not set ENDED. Settlement (Session 4.x) writes it after payment. |
