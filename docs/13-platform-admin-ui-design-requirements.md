# Platform Admin UI Design Requirements

## 1. Goal

本文档用于定义 Restaurant POS 总后台一期的 UI 设计要求，明确：

- 总后台应该呈现什么样的界面风格
- 页面结构和交互应遵循什么原则
- 平台视角和商户视角在 UI 上如何区分
- 各核心页面需要承载哪些信息模块

本文件的目的，是让总后台可以从信息架构阶段进入 wireframe 和视觉设计阶段。

## 2. Design Positioning

总后台不是门店操作界面，也不是商户经营看板。

它应该体现的是：

- 平台管理感
- 治理感
- 稳定性
- 风险可见性
- 配置与控制能力

不要做成：

- 门店收银风格
- 过于轻快的营销后台
- 纯经营分析型 BI 风格

## 3. UI Principles

### 3.1 Platform View First

页面必须优先体现平台视角：

- 商户规模
- 门店状态
- 接入状态
- 风险点

而不是优先体现门店交易细节。

### 3.2 Clear Hierarchy

信息层级要非常清晰：

- 一级：平台总览
- 二级：商户 / 门店 / 终端 / 配置
- 三级：具体状态和操作

### 3.3 Operational Confidence

总后台界面应给人“可控、可治理、可排查”的感觉。

按钮和状态设计应偏：

- 管理
- 巡检
- 配置
- 定位问题

### 3.4 Distinguish Normal vs Risk States Clearly

总后台必须特别强调：

- 正常
- 异常
- 未配置
- 已停用
- 同步失败

状态不能只靠文字，必须有清晰视觉区分。

## 4. Overall Layout Requirements

建议布局：

- 左侧主导航
- 顶部全局栏
- 中间内容区
- 列表 + 详情 / 抽屉为主

建议结构：

- 左侧：一级模块
- 顶部：全局搜索、平台身份、环境状态
- 内容：表格、状态卡、详情面板、监控模块

## 5. Design Tone

建议视觉方向：

- 专业
- 稳定
- 干净
- 偏平台化

建议避免：

- 太像收银端
- 太像顾客端
- 太像电商运营后台

## 6. Core Status Colors

建议状态色语义：

- 绿色：正常 / 已完成 / 在线
- 蓝色：已配置 / 平台管理状态
- 橙色：待处理 / 警告 / 待确认
- 红色：异常 / 失败 / 风险
- 灰色：未启用 / 离线 / 未配置

## 7. Core Pages and UI Requirements

### 7.1 Platform Dashboard

页面目标：

- 让平台管理员一眼了解当前平台健康度

核心模块：

- 商户总数
- 门店总数
- 已开通门店
- 在线终端数
- 异常门店数
- GTO 异常数
- 最近风险事件

设计要求：

- 上方做核心指标卡
- 中间做风险与异常模块
- 下方做近期事件或待处理事项

### 7.2 Merchant List

页面目标：

- 统一查看所有商户

核心模块：

- 商户名称
- 商户状态
- 门店数量
- 接入状态
- 最近更新时间

设计要求：

- 以表格为主
- 支持搜索和筛选
- 状态标签清晰

### 7.3 Merchant Detail

页面目标：

- 查看单个商户平台视角下的完整信息

核心模块：

- 商户基础信息
- 状态
- 门店列表
- 绑定模板
- 接入健康度

设计要求：

- 详情页采用分组卡片
- 可查看但不过度展示门店交易细节

### 7.4 Store List

页面目标：

- 查看平台下所有门店及接入状态

核心模块：

- 门店名称
- 所属商户
- 门店状态
- 终端状态
- GTO 状态

设计要求：

- 适合做状态筛选
- 重点突出“接入异常门店”

### 7.5 Store Detail

页面目标：

- 从平台视角查看门店接入、设备、配置情况

核心模块：

- 门店基础信息
- 终端设备
- 配置状态
- GTO 同步状态
- 风险提示

设计要求：

- 详情应偏运维/配置视图
- 不要做成商户经营分析页

### 7.6 Device List / Detail

页面目标：

- 查看 POS 终端和设备健康状态

核心模块：

- 设备编号
- 所属门店
- 在线状态
- 支付配置状态
- 打印配置状态
- 最后心跳时间

设计要求：

- 重点突出状态
- 支持批量筛选异常设备

### 7.7 Configuration Templates

页面目标：

- 管理配置模板并下发到门店

核心模块：

- 模板名称
- 模板类型
- 使用门店数
- 最近更新时间

设计要求：

- 一期可以先偏表格和表单
- 不需要复杂可视化

### 7.8 Platform Users

页面目标：

- 管理平台内部账号和权限

核心模块：

- 用户名
- 角色
- 状态
- 最近登录

设计要求：

- 重点是权限与状态可见

### 7.9 Support & Monitoring

页面目标：

- 让实施、运维、客服快速找到问题门店和问题终端

核心模块：

- 门店健康监控
- 异常列表
- GTO 失败列表
- 离线设备列表

设计要求：

- 这一页要偏监控风格
- 异常信息要有优先级

## 8. Interaction Requirements

总后台一期建议的交互模式：

- 列表页 + 详情页
- 表格筛选
- 状态标签
- 详情抽屉 / 详情页
- 表单编辑

尽量避免：

- 复杂拖拽
- 大量自由布局
- 重交互配置器

## 9. Relationship to Merchant Admin UI

总后台 UI 必须和商户后台明显区分：

- 总后台：平台治理、接入、配置、异常
- 商户后台：经营、订单、会员、促销、报表

如果一个页面更像门店经营后台，它大概率不应该放到总后台。

## 10. Final Direction

总后台一期 UI 应体现：

- 平台管理感
- 状态治理感
- 配置控制感
- 问题排查感

而不是单纯追求“漂亮后台”。
