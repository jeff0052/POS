# Feature Completion Checklist

## 1. Purpose

本清单用于规定 Restaurant POS 在每完成一个功能后，必须执行的统一检查动作。

目标是避免以下问题：

- 功能“做出来了”，但不符合当前产品模型
- 流程能点通，但状态和数据不稳定
- API、文案、术语与文档不一致
- 新功能影响旧流程，却没有及时发现

本清单适用于：

- Platform Admin
- Merchant Admin
- Store POS
- QR Ordering
- Backend domains

---

## 2. Core Principle

每做完一个功能，不进入下一个功能，直到当前功能完成最小闭环检查。

默认节奏为：

1. 需求检查
2. 流程检查
3. 数据检查
4. 接口检查
5. UI / 交互检查
6. 回归检查

---

## 3. Requirement Check

确认该功能是否仍然符合当前文档定义：

- 是否符合 `Overview`
- 是否符合 `MRD`
- 是否符合 `Architecture`
- 是否符合 `Terminology and State Model`
- 是否符合当前 `Roadmap`
- 是否符合相关 `API Contract`

检查项：

- [ ] 该功能属于当前阶段范围
- [ ] 没有引入未经确认的新业务模型
- [ ] 没有破坏已确定术语
- [ ] 没有绕开状态模型

---

## 4. Flow Check

确认主流程能完整走通，而不是只完成单点页面。

检查项：

- [ ] 用户入口清楚
- [ ] 主要按钮逻辑正确
- [ ] 状态推进符合预期
- [ ] 页面跳转合理
- [ ] 没有出现“功能能点，但不知道当前在做什么”

示例：

- QR 下单后，POS 能否看到
- POS 改单后，刷新是否保留
- cashier 结账后，桌台是否释放

---

## 5. Data Check

确认功能不只是前端假状态，而是有真实数据结果。

检查项：

- [ ] 数据有持久化
- [ ] 刷新后状态仍保留
- [ ] 没有覆盖错误对象
- [ ] 历史数据未被污染
- [ ] 快照字段符合预期

必须重点检查：

- 活动订单是否正确
- 结账是否正确
- 会员与促销数据是否写到正确对象

---

## 6. API Check

确认接口实现与 contract 一致。

检查项：

- [ ] 路径符合约定
- [ ] 字段命名符合术语文档
- [ ] 状态值正确
- [ ] 错误处理明确
- [ ] actor / source 信息可追溯

---

## 7. UI and Interaction Check

确认用户能理解这个功能，而不是只完成视觉布局。

检查项：

- [ ] 页面标题准确
- [ ] 主按钮文案准确
- [ ] 状态文案统一
- [ ] 当前对象（桌、订单、会员等）足够明显
- [ ] 关键反馈存在

---

## 8. Regression Check

确认新功能没有破坏旧功能。

检查项：

- [ ] 旧主流程仍可用
- [ ] 旧状态流仍正确
- [ ] 旧页面没有明显文案冲突
- [ ] 旧接口没有被误改

---

## 9. Acceptance Snapshot

每完成一个功能，至少记录以下内容：

- 功能名称
- 完成范围
- 已验证场景
- 未覆盖风险
- 下一步建议

---

## 10. Recommended Usage

建议每次做完一个功能后，都按以下格式复盘：

### Feature
- 功能名称

### What Changed
- 做了什么

### What Was Verified
- 验证了哪些路径

### Remaining Risk
- 还有什么边界没覆盖

### Next Step
- 下一步做什么

---

## 11. Final Definition

以后本项目默认采用：

**做完一个功能 -> 按 checklist 检查 -> 再进入下一个功能**

而不是：

**连续堆多个功能，最后一起回头补检查。**
