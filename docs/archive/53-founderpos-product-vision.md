# FounderPOS Product Vision

## One Line

**让每一家餐厅都有一个不休息的 AI 合伙人。**

## The Problem

一家餐厅要活下来，老板需要同时是：厨师长、采购员、会计、营销经理、HR、客服、数据分析师。

传统 POS 只解决了"收银"这一个环节。SaaS 工具分散在十几个系统里。老板每天 14 小时，大部分时间不是在做菜，而是在做运营——看报表、想促销、管会员、排班、对账、处理投诉。

**问题不是缺工具，是缺人。**

一个老板不可能同时做好所有事。但他也请不起一个运营总监、一个数据分析师、一个营销经理。

## The Insight

如果每家餐厅有一个 AI，它能：

- 看懂所有经营数据
- 操作所有系统模块
- 对外代表餐厅接洽
- 但关键决策交给老板拍板

那一个人就能管好一家餐厅。甚至十家。

## What FounderPOS Is

FounderPOS 不是一个 POS 系统。它是一个 **AI 驱动的餐厅操作系统**。

```
传统 POS：   老板 → 用软件 → 管餐厅
FounderPOS： 老板 → 审批 AI 方案 → AI 管餐厅
```

### 四层架构

```
Layer 4 ─ Agent Identity
           每家餐厅 = 1 Agent + 1 Wallet
           对外开放：预定 / 包场 / 合作 / 询价
           Agent-to-Agent Protocol

Layer 3 ─ AI Operator
           Restaurant AI Operator（内部运营大脑）
           5 个顾问：菜单 / 营销 / 会员 / 经营 / 出品
           模式：AI 分析 → 生成方案 → 人类审批 → 执行

Layer 2 ─ MCP Tool Layer
           22 个工具覆盖每个业务域
           统一 ActionContext（谁操作 / 为什么 / 谁批准）
           人类和 AI 走同一套工具，审计天然统一

Layer 1 ─ Transaction Foundation
           传统 POS 交易底座
           点餐 / 结账 / 会员 / 促销 / 报表 / 员工 / 班次
           稳定、可审计、独立可用
```

## The Five AI Advisors

每家餐厅的 AI 合伙人有 5 个角色：

| 角色 | 它做什么 | 举例 |
|------|---------|------|
| **菜单顾问** | 分析 SKU 销量、毛利、退菜率、出菜时间 | "这 3 道菜连续亏损 2 周，建议下架" |
| **营销顾问** | 分析客流趋势、促销效果、竞对活动 | "周三客流低 40%，自动生成满 100 减 15 活动" |
| **会员顾问** | 分析消费频次、充值余额、流失风险 | "287 个会员 30 天没来，推一波召回充值" |
| **经营顾问** | 生成日报、周报、毛利分析、异常预警 | "本周毛利下降 5%，主因牛肉成本涨 12%" |
| **出品顾问** | 分析平均出菜时间、退菜原因、高峰瓶颈 | "午高峰出菜超 15 分钟的有 3 道，建议简化" |

### 运作模式

```
Sense  → 通过 MCP Tools 持续感知经营数据
Think  → LLM 分析数据，识别问题和机会
Propose → 生成具体建议方案（草案）
Approve → 推送给老板审批（低风险可自动执行）
Act    → 通过 MCP Tools 执行
```

## Restaurant Agent + Wallet

每家餐厅不只有内部 AI，还有对外身份：

```
Restaurant Agent "老王烧烤"
├── Identity：名称 / 地址 / 菜系 / 风格 / 营业时间
├── Wallet：收款 / 分账 / 结算 / 预算控制
├── Capabilities：可接预定 / 可包场 / 可外卖 / 可团购
└── Protocol：对外暴露哪些交互能力
```

### 对外交互

| 谁来找它 | 交互场景 | 举例 |
|---------|---------|------|
| **顾客 Agent** | 询价 / 订位 / 点餐 | "周六晚上 8 人有位吗？人均预算 150" |
| **企业 Agent** | 包场 / 团建 / 定制 | "30 人团建，预算 5000，有方案吗" |
| **供应商 Agent** | 报价 / 采购 | "明天要 50 斤牛肉，给个价" |
| **平台 Agent** | 上架 / 促销 / 结算 | 美团 / 饿了么 Agent 自动对接 |

### 交互模式

```
外部 Agent → Restaurant Agent → AI Operator 分析
                                     ↓
                              低风险：自动回复（查价、查位）
                              中风险：生成方案 → 老板审批（包场报价）
                              高风险：转人工（大额合作）
```

## Restaurant Agent Network

当每家餐厅都有 Agent + Wallet，真正的杠杆不只是单店自动化，而是餐厅之间形成网络：

### 集体智慧

1000 家餐厅的匿名经营数据，让每家店的 AI 顾问更聪明。

- "同区域同菜系毛利中位数 62%，你只有 55%"
- "你这区域本周外卖单普遍下降 15%，不只是你"
- "类似餐厅做会员充值最有效的方案是充 300 送 50"

### Agent 间协作

- **联合采购**：几家店一起向供应商 Agent 压价
- **互相导流**："我们满了，推荐隔壁同菜系"
- **共享排班**：临时工 Agent 在多家店之间调度

### 一人多店

一个老板通过 FounderPOS 管 10 家店。每家店的 Agent 独立运营，老板只看汇总 Dashboard 审批关键决策。

## RWA: Restaurant as an Asset

每家餐厅有了 Agent + Wallet + 完整经营数据之后，可以做一件传统餐饮做不到的事：

**基于真实经营数据发债。**

```
经营数据（链上可验证）
        ↓
  信用评分模型
        ↓
  发行收益权凭证
        ↓
  投资者购买（获得未来收益分成）
        ↓
  餐厅获得资金（扩店 / 装修 / 采购）
```

- 不是贷款，是基于经营事实的资产证券化
- 不需要银行审批，数据说话
- 投资者看到的不是老板的信用，而是这家店的 AI 验证过的经营表现
- 智能合约自动分润，Wallet 自动执行

## Why This Wins

| 维度 | 传统 POS | 餐饮 SaaS | FounderPOS |
|------|---------|----------|------------|
| 核心价值 | 收银工具 | 管理工具集 | AI 运营合伙人 |
| 用户需要做 | 所有决策 + 所有操作 | 所有决策 + 部分操作 | 只做关键审批 |
| 数据 | 孤岛 | 系统内闭环 | 跨店网络 + 链上可验证 |
| 扩展模式 | 一店一系统 | 一店一账号 | 一人管多店 |
| 商业模式 | 卖硬件/软件 | SaaS 订阅 | 基础免费 + AI 增值 + RWA 抽成 |
| 护城河 | 无 | 弱（可替换） | 网络效应 + 数据飞轮 |

## Phased Delivery

| 阶段 | 内容 | 价值 |
|------|------|------|
| **P0** | MCP Tool Server — 22 个工具 | AI 能操作 POS，是一切基础 |
| **P1** | AI Operator — 5 个顾问角色 | 老板有了 AI 合伙人 |
| **P2** | Agent + Wallet — 餐厅对外身份 | 餐厅能被其他 Agent 找到和交互 |
| **P3** | Agent-to-Agent Protocol | 标准化跨餐厅、跨角色交互 |
| **P4** | Restaurant Network — 匿名数据 | 集体智慧，每家店的 AI 更聪明 |
| **P5** | RWA — 经营数据上链 + 发债 | 餐厅变成可投资的资产 |

**P0-P1 已完成。P2 代码已写完。P3-P5 设计已定，代码待写。**
