## 1. Purpose

定义 FounderPOS 的市场定位、目标客户、核心价值主张、竞争差异化和产品机会。

本文档是对 `02-market-requirements-document.md`（传统 POS MRD）的升级版。传统 POS MRD 聚焦"交易数字化"——统一桌台订单、会员、促销、对账。本文档在此基础上，将产品从**餐饮工具**升级为**餐饮 AI 运营系统**。

## 2. Market Background

### 2.1 餐饮行业现状

中小餐饮（1-20 家门店）是全球最大的线下商业群体之一。以新加坡为例：

- 约 14,000 家餐饮企业，80%+ 是中小型
- 人工成本占营收 30-40%，且持续上涨
- 运营高度依赖老板个人经验和体力
- 数字化程度低——大部分只用收银机 + 基础 POS

### 2.2 市场痛点演变

传统痛点（已被现有 POS 产品部分解决）：
- QR 点单和 POS 点单割裂
- 会员系统过于简单
- 促销规则分散无法追踪
- 商场对账困难

**新一代痛点**（现有产品未解决）：

| 痛点 | 表现 | 影响 |
|------|------|------|
| **老板是瓶颈** | 菜单定价、促销策划、会员运营、排班、对账全靠老板一人 | 开不了分店，休息不了 |
| **经验不可复制** | 什么菜该下架、什么时候做促销、哪个会员快流失，全凭直觉 | 新店/新员工无法复制成功 |
| **数据沉睡** | POS 每天产生大量交易数据，但没有人分析 | 错过优化机会，亏损不自知 |
| **融资困难** | 银行不信任餐饮商户自报的经营数据 | 想扩张但借不到钱 |
| **对外被动** | 客户询价、预定、包场全靠人工回复 | 漏单、响应慢、体验差 |

### 2.3 市场机会

AI 技术（LLM + Agent）的成熟，使得"一个人管好一家店"从幻想变为可能。关键转变：

- LLM 可以理解非结构化的经营问题（"最近生意为什么差了"）
- Agent 可以操作结构化系统（修改促销、调整菜单、生成报表）
- MCP 协议让 AI 能安全地调用任何软件的 API
- 硬件签名（Sunmi POS）可以为经营数据提供不可篡改的背书

**市场窗口：** 目前没有任何餐饮 POS 产品提供"AI Operator"级别的能力。所有竞品的 AI 功能仍停留在"智能推荐菜品"或"自动生成报表"的浅层应用。

## 3. Target Market

### 3.1 Primary Target

**中小餐饮老板（1-5 家门店）**

- 亲自参与运营，是店里的"总经理 + 运营总监 + 财务"
- 有基本的数字化意识（用过 POS、外卖平台）
- 想开分店但人力不够
- 月流水 5-50 万 SGD（或等值当地货币）

### 3.2 Ideal Customer Profile

**新加坡商场中小连锁餐饮**

- 2-5 家门店，使用商米 POS 终端
- 老板 1 人管所有门店，有 1-2 个信任员工
- 已经用过某种 POS（但对运营功能不满意）
- 有扩张意愿但受限于管理能力

### 3.3 Secondary Target

- 东南亚其他地区中小餐饮（马来西亚、印尼、泰国）
- 茶饮、咖啡、轻食、烘焙等细分品类
- 单店创业者（Day 1 就用 AI 运营）

### 3.4 Anti-Target（不做）

- 大型连锁（50+ 门店）——需要定制化 ERP，不是我们的战场
- 纯外卖品牌（无堂食）——我们的核心优势在桌台 + 硬件
- 只需要收银功能的路边摊——客单价太低，无法承载 AI 价值

## 4. Core Value Proposition

### 4.1 一句话

> **FounderPOS 给每家餐厅配一个不休息的 AI 运营总监。老板只管审批。**

### 4.2 价值拆解

| 给谁 | 传统 POS 做到的 | FounderPOS 做到的 |
|------|----------------|------------------|
| **老板** | 查看报表、手动配促销 | AI 每天告诉你该做什么，你点"同意"就行 |
| **会员** | 积分、充值 | AI 自动识别流失风险，生成召回方案 |
| **菜单** | 手动上下架 | AI 分析哪些菜亏钱，建议下架或调价 |
| **促销** | 手动配满减规则 | AI 根据客流趋势自动生成促销草案 |
| **出品** | 看订单做菜 | AI 分析出菜时间，预警出品瓶颈 |
| **财务** | 日结、对账 | AI 自动生成经营摘要，异常一目了然 |
| **融资** | 自己做报表去银行 | POS 硬件签名生成可信经营报告，直接申请贷款/发债 |
| **对外** | 客户打电话问 | Restaurant Agent 自动回复预定、包场询价 |

### 4.3 核心差异化

**不是给餐厅加了一个 AI 功能，是给餐厅配了一个 AI 合伙人。**

| 维度 | 传统 POS（美团/客如云/二维火） | FounderPOS |
|------|-------------------------------|------------|
| 定位 | 交易工具 | 运营系统 |
| AI 角色 | 辅助功能（推荐菜品） | 一等公民 Operator（和老板平级） |
| 操作者 | 只有人 | 人 + AI 双操作者 |
| 决策模式 | 老板看数据 → 自己决策 → 自己操作 | AI 看数据 → AI 生成方案 → 老板审批 → AI 执行 |
| 对外交互 | 被动等客户联系 | Restaurant Agent 主动/自动响应 |
| 融资能力 | 无 | 硬件签名可信报告 → RWA 发债 |
| 扩展性 | 加门店 = 加人 | 加门店 = 加一个 Agent |

## 5. User Scenarios

### 5.1 场景一：老板的早晨

**传统：** 老板 8 点到店，打开后台看昨天报表，花 30 分钟分析数据，手动调整今天的促销，检查库存，安排员工任务。

**FounderPOS：** 老板 8 点打开手机，收到 AI 经营顾问的晨报：
- "昨日营收 ¥12,340，同比 +8%"
- "3 道菜连续 5 天毛利为负，建议下架"（附按钮：同意/稍后）
- "周三历史客流低 30%，已生成满 50 减 10 促销草案"（附按钮：发布/修改/拒绝）
- "会员王女士 45 天未到店，建议推送 20 元召回券"（附按钮：同意/跳过）

老板花 3 分钟点完"同意"，AI 自动执行。

### 5.2 场景二：包场询价

**传统：** 客户打电话问"周六 20 人包场多少钱"，店员记下来告诉老板，老板算价格回复，来回 2-3 天。

**FounderPOS：** 客户的 Agent（或微信/WhatsApp）发消息给 Restaurant Agent：
- "周六 20 人包场，预算 4000"
- Restaurant Agent 自动查询：周六预订情况、食材成本、历史包场均价
- 生成方案推给老板："建议报价 3800，含 10 道菜 + 酒水，毛利预估 42%"
- 老板点"同意"→ Agent 自动回复客户 → 完成预订

### 5.3 场景三：融资扩张

**传统：** 老板想开第二家店，去银行贷款，被要求提供 6 个月财务报表，找会计做报表，银行不信任，审批 3 个月，最终拒绝。

**FounderPOS：**
- POS 硬件每笔交易自动数字签名（不可篡改）
- AI 每月自动生成可信经营报告（硬件签名背书）
- 老板点"申请融资"→ 系统提交报告到 ANext 银行
- 银行可独立验证数据真实性 → 快速审批
- 后续：基于可信数据发行 RWA 短期债，投资人直接投

### 5.4 场景四：一人管多店

**传统：** 老板开了 3 家店，每天在 3 个店之间跑，最多管 5 家就到极限。

**FounderPOS：** 每家店有独立的 AI Operator，自主运营日常事务。老板打开 Dashboard：
- 3 家店的今日营收一目了然
- 红色预警："B 店今日出菜平均慢了 5 分钟"
- 黄色提醒："C 店周末排班不足，AI 建议多排 1 人"
- 待审批 3 项（跨店汇总）

老板花 5 分钟处理完，不需要去任何一家店。

## 6. Product Phases

### Phase 0: Traditional POS（当前进行中）

**交付物：** 能正常使用的传统 POS 功能
- 统一桌台订单（POS + QR 共用一桌一单）
- 基础 CRM（会员、积分、充值、等级）
- 促销规则（满减、满赠、会员价）
- 结账 + 支付（DCS 刷卡、VibeCash QR、现金）
- 商户后台 + 报表 + GTO 对账

**目标：** 先做一个能用的 POS，有基本的商户可以跑起来。

### Phase 1: MCP Tool Server + AI Foundation

**交付物：** 每个域暴露为 MCP Tool，AI 可以调用
- ActionContext 审计贯穿所有操作
- REST API 暴露工具清单和执行能力
- FounderOS Agent 可以连接并操作 POS

**目标：** 打通 AI 操作通道，验证 AI 可以读取和操作每个模块。

### Phase 2: AI Operator

**交付物：** 5 个 AI 顾问角色实际运行
- 菜单顾问、营销顾问、会员顾问、经营顾问、出品顾问
- 每日经营晨报自动推送
- AI 生成建议草案 → 老板审批 → 自动执行
- 风险分级（Low/Medium/High）

**目标：** 老板开始感受到"AI 在帮我管店"。

### Phase 3: Restaurant Agent + Wallet

**交付物：** 每家餐厅的自主身份
- 对外 Agent 身份（可接预定、包场、询价）
- Wallet 集成（收款、分账、预算控制）
- 基础 Agent-to-Agent 交互

**目标：** 餐厅从被动工具变成主动服务方。

### Phase 4: Agent-to-Agent Protocol

**交付物：** 标准化交互协议
- 顾客 Agent ↔ 餐厅 Agent（订位、点餐）
- 供应商 Agent ↔ 餐厅 Agent（报价、采购）
- 平台 Agent ↔ 餐厅 Agent（外卖同步）

**目标：** 建立餐饮 Agent 生态的交互标准。

### Phase 5: Credit Layer (RWA)

**交付物：** 基于经营数据的融资能力
- POS 硬件签名生成可信经营报告
- 信用评分模型
- 通过持牌方发行 RWA 短期债

**目标：** 餐厅可以用经营数据融资扩张。

### Phase 6: Restaurant Agent Network

**交付物：** 多餐厅协作网络
- 匿名经营基准线（同区域/同菜系对比）
- 联合采购、互相导流、共享排班
- 集体智慧让每家店的 AI 更聪明

**目标：** 网络效应 — 用得越多，每家店的 AI 越强。

## 7. Competitive Landscape

### 7.1 直接竞品

| 产品 | 定位 | AI 能力 | 差异 |
|------|------|---------|------|
| **美团收银** | 外卖 + 收银一体 | 基础推荐 | 绑定美团生态，不独立 |
| **客如云（阿里）** | 餐饮 SaaS | 报表分析 | 只有分析没有执行 |
| **二维火** | 收银 + 会员 | 无 | 纯工具，无 AI |
| **Square** | 支付 + POS | 基础分析 | 不做餐饮深度运营 |
| **Toast** | 美国餐饮 POS | 基础自动化 | 贵，不做东南亚 |

### 7.2 竞争优势

没有任何竞品同时具备：

1. **AI 作为一等公民 Operator**（不是附属功能）
2. **MCP 协议 + Agent 架构**（可组合、可扩展）
3. **硬件签名信任层**（POS 数据不可篡改）
4. **RWA 融资能力**（数据变资产）
5. **Agent-to-Agent 生态**（餐厅不再是信息孤岛）

### 7.3 护城河

1. **数据飞轮** — 越多餐厅用 → AI 越聪明 → 更多餐厅用
2. **硬件绑定** — 商米 POS 硬件签名是独家合作
3. **Agent 网络效应** — 餐厅 Agent 越多 → 外部 Agent 越愿意对接 → 更多餐厅加入
4. **FounderOS 生态** — 作为 FounderOS 垂直应用，复用 Office Agent、FPMS 等基础设施

## 8. Business Model

### 8.1 收入结构

| 收入来源 | 定价模型 | 阶段 |
|---------|---------|------|
| **基础 POS** | 免费或低月费（¥99/月） | Phase 0 |
| **AI Operator** | 增值订阅（¥299-499/月/店） | Phase 2 |
| **Agent 服务** | 按交互量收费（预定、包场等） | Phase 3 |
| **RWA 融资** | 融资金额的 1-2% 服务费 | Phase 5 |
| **数据网络** | 高级洞察订阅 | Phase 6 |

### 8.2 定价策略

- Phase 0-1：免费 / 极低价获客，建立用户基础
- Phase 2：AI 价值明确后开始收费，按"省下的人力成本"定价
- Phase 3+：交易抽成 + 增值服务，随使用深度增长

### 8.3 单位经济

**目标 LTV/CAC：**
- 获客成本（CAC）：¥500-1000（通过商米渠道）
- 月均收入（ARPU）：¥400（基础 POS ¥99 + AI ¥299）
- 留存率：90%+（AI 产生粘性后很难换）
- 生命周期价值（LTV）：¥19,200（48 个月 × ¥400）
- LTV/CAC > 19x

## 9. Success Criteria

### Phase 0（当前）
- [ ] 1 家试点餐厅正常使用传统 POS 功能
- [ ] POS + QR 统一桌台订单跑通
- [ ] 支付（DCS + VibeCash + 现金）跑通

### Phase 1
- [ ] FounderOS Agent 可以通过 MCP 调用所有 POS 域
- [ ] 至少 20 个 MCP Tools 可用

### Phase 2
- [ ] 老板收到每日 AI 经营晨报
- [ ] AI 每周生成 ≥3 个可执行建议
- [ ] 老板审批后 AI 自动执行
- [ ] 老板日均花在运营上的时间减少 50%

### Phase 3
- [ ] Restaurant Agent 自动处理预定询价
- [ ] Wallet 日常收支自动对账

### Phase 5
- [ ] 1 家餐厅通过可信报告获得贷款/融资

### Phase 6
- [ ] 10+ 家餐厅加入网络
- [ ] 匿名基准线数据可用

## 10. Risks and Mitigations

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| AI 建议质量差导致老板不信任 | 产品价值归零 | Medium/High 操作必须人类审批；从 Low risk 操作建立信任 |
| 传统 POS 功能不稳定 | 基础体验差，无法推进 AI | Phase 0 必须做到"至少和竞品一样好" |
| 商米合作关系变化 | 失去硬件签名能力 | 架构上不绑死商米；Payment Domain 支持多 provider |
| RWA 合规风险 | 法律问题 | 通过持牌方发行，自己不碰证券 |
| 竞品抄袭 AI 功能 | 差异化减弱 | Agent 网络效应是真正护城河，功能可抄但网络不能 |

## 11. The Assignment

**当前阶段（Phase 0）的唯一目标：做一个能用的传统 POS。**

在这个过程中：
1. 架构上按 AI-ready 设计（ActionContext、MCP Tool 接口预留）
2. 不提前做 AI 功能，但确保后续加入时不需要推翻底座
3. 支付集成（DCS + VibeCash）是当前最大的技术风险，优先解决

Phase 0 做完、有 1 家餐厅跑起来之后，立刻进入 Phase 1（MCP Tool Server，代码已写完待合并）。
