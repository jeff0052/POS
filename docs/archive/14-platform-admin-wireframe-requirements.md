# Platform Admin Wireframe Requirements

## 1. Goal

本文档用于定义 Restaurant POS 总后台一期的低保真线框要求，明确：

- 每个核心页面应包含哪些区块
- 信息顺序如何组织
- 哪些内容需要固定显示
- 哪些操作属于主操作、次操作

本文件用于连接：

- Platform Admin Requirements
- Platform Admin Information Architecture
- Platform Admin UI Design Requirements

让总后台可以直接进入 wireframe 设计阶段。

## 2. Wireframe Principles

总后台 wireframe 应遵循：

- 先信息结构，后视觉风格
- 先管理效率，后装饰效果
- 先状态可见，后数据丰富
- 先平台治理视角，后业务细节

## 3. Global Layout Wireframe

所有总后台核心页面建议采用统一骨架：

- 顶部全局栏
- 左侧主导航
- 页面标题区
- 页面操作区
- 主内容区

### 3.1 Top Bar

建议包含：

- 平台名称 / Logo
- 当前环境标识
- 全局搜索（后续可选）
- 当前登录人

### 3.2 Left Navigation

建议固定一级菜单：

- Dashboard
- Merchants
- Stores
- Devices
- Configurations
- Platform Users
- Support & Monitoring

### 3.3 Page Header

每页应包含：

- 页面标题
- 页面说明（可选）
- 页面级主操作按钮

## 4. Page-by-Page Wireframe Requirements

### 4.1 Platform Dashboard

页面目标：

- 一眼看到平台健康度

页面区块：

1. 顶部核心指标卡
   - 商户数
   - 门店数
   - 在线终端数
   - 异常门店数
2. 异常概览区
   - 离线终端
   - GTO 异常
   - 未完成接入门店
3. 最近事件区
   - 最近异常
   - 最近配置变更

主操作：

- 刷新
- 查看异常

### 4.2 Merchant List

页面目标：

- 快速查找和管理商户

页面区块：

1. 页面标题区
2. 筛选区
   - 搜索框
   - 状态筛选
3. 商户表格区
   - 商户名
   - 状态
   - 门店数
   - 接入状态
   - 更新时间
4. 分页区

主操作：

- 新建商户

次操作：

- 查看详情
- 编辑
- 停用 / 启用

### 4.3 Merchant Detail

页面目标：

- 查看单个商户的平台接入情况

页面区块：

1. 基础信息卡
2. 状态卡
3. 所属门店列表
4. 配置模板信息
5. 风险 / 接入状态摘要

主操作：

- 编辑商户

### 4.4 Store List

页面目标：

- 从平台视角查看所有门店

页面区块：

1. 标题区
2. 筛选区
   - 商户筛选
   - 状态筛选
   - 接入状态筛选
3. 门店表格区
   - 门店名
   - 商户
   - 状态
   - 终端状态
   - GTO 状态
4. 分页区

主操作：

- 新建门店

### 4.5 Store Detail

页面目标：

- 查看门店的接入、设备、配置状态

页面区块：

1. 门店基础信息
2. 接入状态卡
3. 设备列表
4. 配置状态卡
5. GTO 状态卡
6. 风险提醒区

主操作：

- 编辑门店
- 查看终端

### 4.6 Device List

页面目标：

- 管理和定位终端状态

页面区块：

1. 标题区
2. 筛选区
   - 在线状态
   - 门店筛选
   - 设备类型筛选
3. 设备表格区
   - 设备编号
   - 所属门店
   - 在线状态
   - 支付配置状态
   - 打印配置状态
   - 最后心跳

主操作：

- 刷新状态

### 4.7 Device Detail

页面目标：

- 查看终端级问题和配置状态

页面区块：

1. 基础信息卡
2. 在线状态卡
3. 支付 / 打印配置状态卡
4. 最近错误或异常记录

### 4.8 Configuration Templates

页面目标：

- 管理平台模板并下发

页面区块：

1. 模板列表
2. 模板详情 / 编辑表单
3. 关联门店列表

主操作：

- 新建模板
- 下发模板

### 4.9 Platform User List

页面目标：

- 管理平台内部人员

页面区块：

1. 用户表格
   - 用户名
   - 角色
   - 状态
   - 最近登录
2. 筛选区
3. 分页区

主操作：

- 新建平台账号

### 4.10 Support & Monitoring

页面目标：

- 快速发现并定位问题门店

页面区块：

1. 异常指标概览
2. 异常门店列表
3. 离线终端列表
4. GTO 失败列表
5. 最近告警区

主操作：

- 查看门店详情
- 查看终端详情
- 重试 GTO

## 5. Action Hierarchy

总后台按钮层级建议：

### 主操作

- 新建
- 下发配置
- 刷新状态

### 次操作

- 编辑
- 查看详情
- 启用 / 停用

### 风险操作

- 停用商户
- 停用门店
- 重置终端

风险操作必须视觉上更克制，并带确认。

## 6. Status Display Requirements

wireframe 阶段就要预留状态标签位置。

重点状态包括：

- 商户状态
- 门店状态
- 接入状态
- 终端在线状态
- 配置状态
- GTO 状态

## 7. Final Direction

总后台 wireframe 的核心，不是复杂图形，而是：

- 平台对象清晰
- 状态清晰
- 页面职责清晰
- 支持排查和管理

如果一张页面在 wireframe 阶段就看不出“平台治理感”，那它的结构大概率还不够好。
